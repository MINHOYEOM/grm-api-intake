## 4. Phase D — 누출 경로 색출 (근본원인)

**질문: 스캐폴드가 만들지 않는 `m_99/m_218 view.do?seq=` 가 어떻게 발행본에 들어갔나?**

1. **scaffold 는 admin/recall 에 m_XX 를 만들지 않는다**(§1 #9/#10 은 nedrug/data.go.kr). 그러나 **RSS 경로(#8)는 official_url=RSS `<link>`=`m_XX/view.do?seq=` 를 정당하게 통과**시킨다. 즉 m_XX 링크 자체는 **handoff 근거가 있으면 정상**이다.
2. 결정적 비대칭: **`m_218`(안내서/지침)은 수집기 보드(data0013)** 라 handoff 에 실릴 수 있지만, **`m_99`(보도자료)는 어떤 수집기 RSS 보드도 구독하지 않는다.** → 발행본의 `m_99` 링크는 **handoff 근거가 원천적으로 없다 = 날조.**
3. LLM 이 URL 을 직접 쓰는 통로(코드 방어 부재):
   - **검색 카드 미니 템플릿**(v16 L346): `📎 공식원본 [링크]({기관 공식 URL})` — LLM 이 "공식 URL" 을 작성.
   - **🔮 Watch 표**(L361): `{링크}` LLM 작성.
   - **graceful degradation**(L151): handoff 부재 시 전 카드 LLM 작성.
   - 방어 = 소프트 규칙 L348("실제 확인한 URL 만·패턴 유추 금지") 1줄뿐.
4. **MFDS 는 Core 검색 슬롯이 없다**(L187: "Intake 흡수가 유일 경로"). 즉 정상 운영이라면 MFDS 카드는 전부 Intake(scaffold) → 링크는 handoff 근거 필수. **그런데 LLM 이 검색/기억으로 보도자료(m_99) URL 을 만들어 MFDS 카드에 넣었다 → L187·L348 동시 위반이 W24 사고.**

> 메모리 정합: 페이지 M2/M3 메타 블록이 LLM 작성이라 코드 중립화(`_neutralize_forbidden`)를 우회한 것과 **동형 구조** — "LLM 자유 작성 영역엔 코드 게이트가 없다". 출처 링크도 같은 사각지대였다.

**재현 가능한 누출 시나리오(우선순위순):** ① 검색 카드 📎 날조(MFDS 는 애초에 검색 카드 금지인데 생성) ② Intake 카드 footer 링크를 LLM 이 치환(Lint 2 위반) ③ graceful 모드 전면 LLM 작성. 셋 다 **"발행물 링크 ⊄ handoff 근거"** 라는 한 가지 불변식 위반으로 수렴 → E1 이 정확히 이걸 막는다.
