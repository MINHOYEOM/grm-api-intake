## 0. 배경 — Phase 3 가 답해야 했던 질문

Phase 1 설계(§4 Phase 3)는 **"Core 8 슬롯 1·3·4·5·6 은 수집기가 이미 커버 → 검색을 cross-check 로 축소(−20~−35 줄), 7 ICH·8 TGA·graceful 만 검색 유일경로"** 를 가설로 세웠다. 또 06-17 dry-run 의 **Deep Dive WebFetch 5/5 HTTP 403**(PL-8) 을 근거로 "수집기 이관 최우선 후보"로 지목했다.

이 가설은 **"검색 = 수집기와 중복된 낭비"** 라는 전제에 기댄다. 그 전제를 **프롬프트 편집 전에** 적대적으로 검증해야 했다 — 틀리면 슬림이 규제 커버리지를 **침묵 누락**시키기 때문이다. 그래서 file:line 증거 기반 10-agent 검증을 먼저 돌렸다.

## 2. 전략 전환 — "제거" → "재구성"

검증 결론: **검색/Fetch 슬롯 어느 것도 안전하게 삭제할 수 없다**(전부 무음 실패 catch 용 load-bearing). Phase 1 설계의 "−20~−35 줄" 추정은 **삭제 가능 전제 위에서 나온 것**이라 성립하지 않는다.

그러나 지시문의 **진짜 목표**는 줄 수가 아니라 **"오류 표면(LLM 자유작성 구간) 자체를 줄이는"** 것이다(가짜-URL 사고의 근본). 그 목표는 **삭제 없이도** 달성된다 — LLM 에게 "수집기가 Master 다·자유 검색/출처 URL 작성 금지·403 은 정상" 을 명시하면 **자유 URL 작성 표면이 줄어든다**(커버리지/cross-check 은 유지·강화). 이것이 본 Phase 3 의 편집이다.
