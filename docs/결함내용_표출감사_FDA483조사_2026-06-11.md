## 0. 한 줄 결론

GRM 은 **결함 내용을 MFDS GMP 실태조사(P6)에서만 제대로 추출**하고, 정작 글로벌에서 가장 풍부한 두 결함 원본 — **WHO WHOPIR(공개 실사보고서)** 과 **FDA Warning Letter** — 은 **링크와 회사명만 잡고 결함 본문은 버리고 있다.** 즉 "왜·무엇을"이 빠진 표면 카드가 나오는 구조적 원인이 확인됐다. **가장 레버리지 큰 개선 = 새 소스 추가가 아니라, P6 의 "문서 fetch + findings excerpt" 패턴을 WHOPIR·WL 로 확장(Track A).** 추가로 **FDA 483/EIR 은 예상보다 수집 가능**(OII FOIA Reading Room = 구조화 표 + RSS + XLSX + 건별 PDF)하며 가장 깊은 결함 원본이라 신규 소스 1순위(Track C).

## Track A — 이미 수집 중인 소스의 결함 표출 감사 (코드 확인)

각 수집기가 **결함 본문(why)을 raw 에 캡처하는지**, 그게 **prose_input → 카드 슬롯(W5/W6/W7)** 에 흐르는지 코드로 확인.

| 소스 | 결함 본문 캡처? | 실제 캡처 내용 | 결함 "why" 갭 |
|---|---|---|---|
| **MFDS GMP 실태조사** (P6, `collect_mfds_gmp_inspection.py`) | ✅ **예** | `attachment_text`·`attachment_deficiency_excerpt`(표지 이후 지적사항 findings 구간) | **없음 — 이게 모범 사례·재사용할 엔진** |
| **WHO WHOPIR** (`collect_who.py::_collect_whopir`) | ❌ **아니오** | PDF **링크**(abs_url) + 제조소명(manuf) + `anchor_text`(링크 텍스트뿐). **PDF 본문을 fetch/parse 안 함** | 실사보고서 결함 전체가 시스템에 안 들어옴 → 카드는 "WHO가 사이트 X 실사보고서 냈다"까지만 |
| **FDA Warning Letter** (`collect_intake.py::collect_fda_warning_letters`) | ❌ **아니오** | 목록 페이지 메타(회사·날짜·부서·subject·`wl_href`)만. **편지 본문(위반 내용)을 fetch 안 함** | cGMP 위반 서술이 `wl_href` 에 있는데 안 염 → 카드는 제목/회사까지만 |
| OpenFDA Recall | △ 부분 | `reason_for_recall`(짧은 사유) | 사유 한 줄 — 상세 결함 아님(소스 한계) |
| HC Recall | △ 부분 | `Issue`·`What you should do` | 중간 깊이 |

**근거(prose_input 매핑, `card_scaffold.py::_prose_input`):** `issue_or_reason` 폴백 체인 = `RTRVL_RESN(회수사유)·reason_for_recall·Issue·attachment_deficiency_excerpt·attachment_text·…·anchor_text·description`. → **GMP(attachment_text)·recall(사유)** 만 결함성 텍스트를 갖고, **WHOPIR 은 anchor_text(링크 텍스트), WL 은 본문 필드 자체가 없음.** 즉 LLM 이 받는 컨텍스트에 두 소스의 "why"가 애초에 부재.

### A. 결론·강화 방향
- **새 소스가 아니라 추출 깊이가 문제.** raw 는 이미 들어오는데(WHOPIR PDF 링크·WL 링크) 문서를 안 열어 결함을 버린다.
- **고레버리지 순서:**
  1. **WHOPIR 결함 추출**(최우선): `_collect_whopir` 가 PDF(abs_url)를 fetch → PyMuPDF(이미 의존성)로 결함/결론 구간 excerpt → `raw_payload`에 `whopir_excerpt` 추가. **P6 의 PDF findings 추출 로직 재사용.** 이미 100% 버리던 것을 살리는 거라 효과 최대.
  2. **FDA WL 본문 추출**: `collect_fda_warning_letters` 가 `wl_href`(편지 페이지) HTML 본문을 fetch → 위반 서술(주로 "Violation"/"During our inspection" 단락) excerpt → `wl_raw["wl_body_excerpt"]`. HTML 파싱(PyMuPDF 불요).
  3. prose_input 폴백 체인에 `whopir_excerpt`·`wl_body_excerpt` 추가 → 카드 W5/W6/W7 이 결함 내용을 반영.
- **주의:** WHOPIR/WL 문서 fetch 는 **수집기(GitHub Actions Python)** 가 하므로 Routine WebFetch 와 무관(사용자 요구 충족). 단 건별 fetch 가 늘어 수집 시간↑·실패 graceful degrade 필요(P6 의 manual-review 플래그 패턴 재사용). EVAL-1 의 E4(점검 실행가능성)·E6(페르소나)가 강화 효과를 측정하는 짝.

## Track C — FDA 483 / EIR 타당성 조사 (라이브 fetch)

### C.1 무엇 (가장 깊은 결함 원본)
- **Form FDA 483** = FDA 실사관이 실사 종료 시 발부하는 **관찰사항(inspectional observations)** 목록 — "무엇을 어떻게 잘못했는지"의 *원본*. (예: 무균공정 결함·데이터 무결성·CAPA 미흡 등 항목별 서술)
- **EIR(Establishment Inspection Report)** = 실사 종합보고서.

### C.2 접근성 (라이브 확인 — OII FOIA Electronic Reading Room)
- **URL**: `https://www.fda.gov/about-fda/office-inspections-and-investigations/oii-foia-electronic-reading-room`
- **구조화 표**(fetch 가능·차단 없음): `Record Date · Company Name · FEI Number · Record Type · State · Country · Establishment Type · Publish Date · Excerpt`.
- **Record Type 필터**에 `483` · `Establishment Inspection Report (EIR)` · `483 Response` · `Amended 483` 등.
- **건별 483 PDF 직링크**(예: `fda.gov/media/192439/download`) → 관찰사항 원문.
- **XLSX 일괄 다운로드** + **RSS 피드**(`.../rss-feeds/ora-foia-electronic-reading-room/rss.xml`) 제공 → 증분 감지 용이.
- **현행성**: "Content current as of 05/27/2026" — 활발히 갱신. 최근 항목에 제약 제조소 다수(예: **Eugia Pharma(인도, Sterile Pharmaceutical Manufacturer)**, BPI Labs, SCA Pharmaceuticals 등).

### C.3 GRM 매핑·가치
- `document_id`(dedup) = FEI Number + Record Type + Record Date(또는 media id). `firm` = Company Name · `Site Country` = Country/State · `Establishment Type`(예: Sterile Pharmaceutical Manufacturer) → **무균/제형 신호 직접** · `date` = Publish Date(윈도우).
- `Signal Tier` = **Tier 3**(483 = 미시정 시 WL/집행으로 이어지는 선행 신호). `Evidence` = **483 PDF excerpt = A 후보**(관찰사항 직접 인용).
- **사용자 기준에 가장 부합** — 483 PDF 에 "무엇을 어떻게"가 항목별로 있음.

### C.4 난이도·한계 (솔직)
- **난이도 중상.** 신규 수집기 `collect_fda_483.py`(`ENABLE_FDA_483`, 기본 off). RSS/표 폴링 → 신규 483/EIR → PDF fetch → PyMuPDF 관찰사항 excerpt(P6 재사용). XLSX 경로도 가능.
- **한계 = 부분 커버리지**: Reading Room 은 "select(공개 결정·빈출 요청)" 건만 게시 — **전수 아님**. 다수가 미국 outsourcing/compounding 시설(무균조제), 일부 해외 제약사. → QA 관련성 필터(제조 유형·modality)·노이즈 게이트 필요.
- 483 PDF 는 부분 **리닥션** 가능. 갱신은 publish date 기준 backfill(P6 enforcement 윈도우 패턴).

## 종합 권고

세 작업이 **같은 엔진**(문서 fetch + 결함 excerpt, P6 가 이미 입증)으로 수렴한다. 사용자 "why" 기준에 모두 부합. 우선순위:

| 순위 | 작업 | 유형 | 근거 |
|---|---|---|---|
| **1** | **WHOPIR 결함 추출**(Track A-1) | 기존 강화·새 소스 0 | 이미 pull 하는 PDF 를 100% 버림 → 최고 레버리지. P6 PDF 로직 재사용 |
| **2** | **FDA WL 본문 추출**(Track A-2) | 기존 강화·새 소스 0 | 이미 pull 하는 링크의 위반 본문 미수집. HTML 파싱 |
| **3** | **FDA 483/EIR 신규 수집기**(Track C) | 신규 소스 | 가장 깊은 결함 원본·수집 가능(표+RSS+PDF). 단 부분 커버리지 |

- **구현은 모두 Claude Code 트랙**(수집기 .py + 회귀 + dry-run 게이트). Cowork 는 각 CC 지시문 작성.
- **K3/BIO-1 관찰과 직교**(수집기 additive·scaffold/v16/golden 무관) — P6~P8·GAP-2 선례. 단 운영 활성은 한 소스씩.
- EVAL-1 의 E4·E6 이 "결함이 카드에 실제로 표출돼 실무 점검으로 연결되는가"를 측정하는 검증 짝.

### 드롭/보류 (이전 제안 정정)
- **EDQM CEP = 드롭.** 물질+상태+사유카테고리뿐, 결함 내용 없음 → 사용자 기준 미달.
- **EudraGMDP NCR = 보류.** 상세가 세션 게이트 + "카테고리 수준 why"(결함 영역명·조치)까지 — 483/WHOPIR 보다 얕음. 카테고리 신호로 충분할 때만.
