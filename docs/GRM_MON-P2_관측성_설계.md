# GRM MON-P2 — 상태저장형 관측성 설계문서

## 1. 왜 필요한가 (문제 정의)

### 1.1 P1 health 의 구조적 공백 — stateless

운영 모니터링 P1(§3.5)은 `collect_intake.py::_evaluate_health()` 단일 판정으로, exit code·step summary·`grm-health.json`·Issue 본문이 공유하는 단일 기준이다. 견고하지만 **그날 그 run 하나만** 본다 — stateless.

따라서 **상태(이력)를 저장해야만 잡히는 이상**은 P1 이 원리적으로 못 잡는다:

| 못 잡는 이상 | 왜 stateless 가 못 잡나 |
|---|---|
| 소스 A 가 **연속 N일 0건** | 하루 0건은 §3.5 "0건 판정 원칙"상 정상일 수 있어 그날 run 은 ok. 연속성은 이력이 있어야 보임 |
| 소스 B 의 **침묵 추세**(평소 주 3건 → 2주째 0건) | 각 run 은 "오늘 0건=정상"으로만 판정. baseline 대비 이탈은 누적 비교가 필요 |
| **workflow 자체가 며칠 안 돌았는데 아무도 모름** | run 이 없으면 health JSON 도 안 생김 → 부재의 부재는 그 자리에서 안 보임 |

### 1.2 이미 한 번 데인 전례 — WHO NOC

정밀검토 배치4(B4)에서 WHO NOC 침묵이 health 에 **묻혔던** 사실이 있다. 당시 직접 원인은 NOC `core=False` 라 RSS/WHOPIR 정상 시 NOC 오류가 health 에 안 떴던 **구조적 셀렉터 결함**이었고, B4 가 별칭 선택자+구조 sentinel+core 승격으로 P1 에서 잡도록 고쳤다. 즉 그 *특정* 셀렉터 케이스는 이제 P1 이 잡는다.

MON-P2 가 더하는 것은 그와 **직교하는 추세 차원**이다 — 셀렉터가 멀쩡한데도 소스가 baseline 대비 조용해지는 경우, 그리고 P1 sentinel 이 미처 못 거른 미래의 침묵에 대한 **2차 방어선**(defense-in-depth). 셀렉터 결함의 즉시 탐지는 여전히 P1 의 일이고, MON-P2 는 그 위에서 "추세가 멎었나"를 본다.

### 1.3 지금 가치가 커진 이유

WHY-1 로 수집 소스가 늘었다(FDA 483/EIR·WHOPIR PDF·WL 본문 excerpt). 소스가 많아질수록 "어느 하나가 조용히 멎는" 표면적이 넓어지고, 사람이 그걸 일일이 추적하기는 비현실적이다 → 소스별 침묵 감지의 효용 ↑.

## 2. 설계 원칙

1. **보조 요약자 — P1 비대체.** MON-P2 는 `_evaluate_health` 를 대체하지 않는다. 핵심 즉시 알림(insert 실패·핵심 소스 동시 실패 등)은 수집 직후 같은 workflow 에서 P1 이 판정한다(§3.5). MON-P2 는 그 위에 얹는 **주간 추세 요약자**다. 둘의 책임이 겹치지 않는다.
2. **비파괴·관찰 직교.** 수집 파이프라인·`card_scaffold`·v16 프롬프트·golden 을 일절 건드리지 않는다(P6~P8·GAP-2·WHY-1·EVAL-1 선례와 동일). MON-P2 는 **읽기 전용 관측 계층** — 데이터를 읽기만 하고 쓰지 않는다(Notion row Status 도 불가침). K3 4주·BIO-1 관찰 불훼손.
3. **결정론·재현성.** 판정 입력은 **고정 윈도우의 날짜별 소스 카운트 시계열**이다. run 순서·실행 시점에 흔들리지 않게, 윈도우는 "기준일 KST 자정 직전 N일 폐구간"으로 고정한다. 같은 데이터·같은 기준일 → 같은 판정.
4. **과알림 금지 (최우선 비기능 요건).** 저빈도라 정상적으로 0건인 소스(MHRA·PIC/S·WHO NOC·FDA 483·MFDS Recall/Admin 등)를 이상으로 올리지 않는다. 두 겹의 방지 장치 — ① **소스별 빈도 등급**(저빈도는 절대 0건으로 알리지 않고 추세 이탈만 본다, §4) ② **flag gate**(그 윈도우에 비활성이던 소스는 침묵 판정 제외, §5.3). §3.5 "0건 판정 원칙"을 그대로 존중한다.
5. **코드 0 우선(M1).** EVAL-1 의 E1 단계처럼, 1단계는 Notion MCP 가 연결된 독립 세션 + 실행프롬프트로 **코드 없이** 시작한다. 결정론이 안정되면 그때 Python 으로 내린다(M2, §6).

## 3. 확정 설계 결정 4건

> 사용자가 4건 모두 권장안 채택을 위임(2026-06-12). 각 결정의 근거를 남긴다.

### 결정 1 — 상태저장 위치: **ⓑ Notion Intake DB 이력(주) + ⓒ Run 이력(보조)**

**채택.** 소스별 일별 카운트의 1차 진실원천은 Notion Intake DB 다. 보조로 GitHub Actions run 이력을 본다.

**왜.** 현황 조사에서 드러난 결정적 사실: **`grm-health.json` 은 `.gitignore` 되어 있고 Actions artifact 업로드 step 도 없다 → run 간 보존이 0이다.** 즉 ⓐ(health JSON 이력 누적)는 *새 보존 인프라*(커밋 또는 artifact retention)를 P1 workflow 에 추가해야 하고, 그 자체가 P1 무간섭 원칙과 마찰한다. 반면 Notion Intake DB 는 `Run Date (KST)`·`Source`·`Status` 속성으로 **이미 일별 영구 보존**되어 있어, 새 저장소 0으로 소스별 시계열을 얻는다. §5.3 도 ⓑ/ⓒ 를 시사한다.

**역할 분담.**
- **ⓑ Notion (주):** `Run Date (KST)` × `Source` 로 집계 → 소스별 일별 *적재 건수* 시계열. "이 소스가 최근 며칠간 새 row 를 넣었나"의 직접 신호.
- **ⓒ Run 이력 (보조):** GitHub Actions 의 최근 run 성공/실패/warning + 기존 운영 경고 Issue. "workflow 가 실제로 돌긴 했나"를 교차확인 — 0건이 *소스 침묵*인지 *workflow 미실행*인지 구분하는 데 필수.

**한계(정직히 명시).** Notion 은 *적재된* row 만 본다 — `fetched>0 이지만 전부 dedup-skip → inserted=0` 인 경우와 *진짜 0 fetched* 를 구분하지 못한다. 저빈도 소스에서 이 둘 다 "0 row"로 보인다. 이 한계의 보정은 (a) ⓒ run-level 성공 교차확인, (b) M2 단계에서 health JSON 을 artifact 로 보존해 `fetched`/`inserted` 구분을 입력에 추가(§6·§9). M1 은 이 한계를 안고 가되 추세 차원에서는 충분히 동작한다.

### 결정 2 — baseline 정의: **소스별 빈도 등급 + 등급별 임계**

**채택.** 각 소스를 high/med/low 빈도로 분류하고 등급별로 연속 0건 임계를 달리 적용한다(상세 표 §4). 저빈도 등급은 **절대 0건으로 알리지 않고** 소스 자신의 직전 baseline 대비 이탈만 본다.

**왜.** 과알림 방지가 핵심 요건이다. "전 소스 연속 7일 0건" 단일 임계는 §3.5 가 예시로 들었지만 그대로 적용하면 저빈도 소스(정상적으로 2~3주 0건)에서 과알림이 다발한다. 빈도 등급은 GAP-2 의 큐레이티드 사전 선례처럼 **근거 있는 시드값**으로 시작해 관찰로 보정한다(동적 baseline 은 M2 에서 누적 데이터로 자동 산출, §6).

### 결정 3 — 실행 주체·주기: **Cowork 스케줄드 태스크 (주 1회)**

**채택.** EVAL-1 선례대로 독립 세션의 Cowork 스케줄드 태스크가 주 1회 실행프롬프트를 돌려 Notion/Actions 이력을 읽고 요약·판정한다.

**왜.** ① P1 workflow 를 *전혀* 건드리지 않는다(주간 Actions step 안은 P1 무간섭 원칙과 마찰). ② 읽기 전용·독립 세션이라 관찰 직교가 자연스럽다. ③ EVAL-1 과 동일 운영 모델이라 운영자 인지부하가 낮다(둘 다 "독립 세션 보조 요약자" 짝). ④ M1 은 코드 0 — 스케줄드 태스크 + 프롬프트만으로 성립. "별도 Codex heartbeat"는 명세 용어와 일치하나 Codex 실행 인프라를 따로 세워야 해 ROI 가 낮다.

### 결정 4 — 알림 경로: **기존 GitHub Issue 재사용**

**채택.** MON-P2 이상 알림은 기존 고정 제목 `GRM Intake 운영 경고` Issue 에 댓글 누적하거나, 동일 메커니즘의 전용 라벨 Issue 로 emit. 주간 요약 스코어카드는 repo 문서/Cowork 아티팩트로 누적.

**왜.** 단일 사용자 단계에서 새 채널(Slack·이메일)은 효용이 낮다. 기존 운영 경고 Issue 누적 메커니즘이 이미 있어 추가 인프라 0. 채널 다변화는 다사용자 전이 시 별도 P3 로 명시 보류(우선순위 문서·§5.2 와 정합).

**"추세 이탈만"의 의미.** 저빈도 등급은 단순 0건으로 알리지 않는다. 대신 *그 소스 자신의 직전 적재 간격*을 baseline 으로 삼아, **현재 침묵 길이가 직전 baseline 간격을 유의하게 초과**(예: ×2 또는 임계 일수 도달 중 더 긴 쪽)할 때만 INFO/WARN 으로 보고한다. 데이터가 적어 baseline 산출이 불가하면 "관측 부족"으로 표기하고 알림하지 않는다(과알림보다 미알림 우선).

**비활성 소스.** 그 윈도우에 flag off 였던 소스(현재 schedule 기본은 ICH 등 일부만 on)는 0건이 당연하므로 **침묵 판정에서 제외**한다(§5.3 flag gate).

## 5. 침묵 감지 알고리즘 (결정론·재현성)

### 5.1 입력 윈도우 (고정)

- 기준일 `D` = 실행일 KST. 윈도우 = **`[D-W+1, D]` 폐구간**, 기본 `W=7`(주간 요약). 단 저빈도 추세 판정은 더 긴 lookback `L`(기본 30~45일)이 필요하므로, 조회는 `max(W, 등급별 임계)` 까지 확장하되 **요약 표시 윈도우는 W=7 로 고정**(조회/표시 분리 — B1 선례와 동일 패턴).
- 윈도우 경계가 날짜로 고정되므로 같은 `D` 면 언제 돌려도 같은 입력 → 재현 가능.

### 5.2 판정 절차 (소스별)

1. Notion Intake DB 에서 `Run Date (KST) ∈ [D-L+1, D]` 인 row 를 `Source` 로 group-by → 소스별 일별 적재 카운트 시계열을 만든다. (`GRM Handoff` Source 는 입력 큐라 제외.)
2. 소스가 그 윈도우에 **활성**이었는지 확인(§5.3). 비활성이면 skip.
3. **High/Med 등급:** 마지막 비-0 적재일부터 `D` 까지의 연속 0-적재 일수를 센다. 등급 임계 초과 → **침묵 후보**.
4. **Low 등급:** 직전 baseline 적재 간격(최근 비-0 적재 간 간격의 중앙값)을 구하고, 현재 침묵 길이가 `max(직전 baseline ×2, 등급 임계)` 초과 → **추세 이탈 후보**. baseline 산출 불가 → "관측 부족", 알림 없음.
5. 모든 침묵/이탈 후보를 **ⓒ run-level 로 교차확인**: 같은 윈도우에 Actions run 이 정상 실행됐는가?
   - run 이 며칠 없었다 → 원인은 *소스 침묵이 아니라 workflow 미실행* → 별도 분류 "WORKFLOW_GAP"(소스별 침묵으로 오귀속 금지).
   - run 은 정상인데 소스만 0 → 진짜 침묵 후보로 확정.

### 5.3 flag gate (과알림 방지의 핵심)

소스의 침묵 판정은 **그 윈도우에 그 소스가 활성(flag on)이었을 때만** 한다.

- M1(현황): 현재 활성 소스 집합을 실행프롬프트에 **명시 상수**로 둔다(repo workflow `vars.ENABLE_*` 현행값과 정합; 변경 시 프롬프트 동기화). 현행 schedule 기본 활성/비활성 구성을 반영.
- 교차 단서: Notion 에 그 소스 row 가 과거엔 있었는데 최근 끊겼다 = 활성이었던 정황. 과거에도 전무 = 미활성/미수집 가능성 → 보수적으로 "관측 부족" 처리.
- M2(후속): health JSON artifact 의 `flags`/`active_sources` 를 persisted 입력으로 받아 flag gate 를 자동화(§6·§9). M1 의 수동 상수 한계를 흡수.

### 5.4 한계와 보정 (정직히)

- **dedup-skip 맹점**(§3 결정1): Notion 은 inserted 만 본다. `fetched>0·전부 dedup → inserted=0` 을 침묵으로 오인할 수 있다. M1 은 추세 차원에서만 신뢰하고, 단발 0건은 약신호로 취급. M2 에서 `fetched` 분리로 해소.
- **셀렉터 결함 즉시 탐지는 P1 의 일.** MON-P2 는 추세·2차 방어선이지 1차 구조 검출기가 아니다(B4 sentinel 역할 분담).

## 6. 아키텍처 — 2단계 (EVAL-1 과 동일 패턴)

### M1 단계 (지금): 프롬프트 + Cowork 스케줄드 태스크 — 코드 0
주 1회 스케줄드 태스크가 Notion MCP 연결 독립 세션에서 `prompts/GRM_MON-P2_주간점검_실행프롬프트.md` 를 실행 → Notion 이력 집계 + Actions run 교차확인 → §5 판정 → 주간 스코어카드 + 이상 알림(이상 시에만 Issue).
- **장점:** 코드 0줄, 즉시 가동, P1 workflow 무간섭. 수집 파이프라인·golden 불변.
- **산출:** 매주 같은 포맷 스코어카드(§7) → 누적이 M2 의 동적 baseline 입력이 된다.

### M2 단계 (관찰 후, K3 종료 이후 권장): Python 반자동화
M1 을 4~6주 돌려 등급·임계가 안정되면 결정론 부분을 Python 으로 내린다.
- **결정론 영역(Python):** Notion 쿼리·일별 카운트·연속 0건·동적 baseline 간격 산출·WORKFLOW_GAP 분류.
- **판단 영역(잔류):** 등급 경계 케이스 해석·신규 소스 등급 배정.
- **보존 보강:** 이 단계에서 P1 workflow 에 **health JSON artifact 업로드 step**(또는 별도 history 파일)을 더해 `fetched`/`inserted`/`flags` 를 persisted 입력으로 확보 → §5.4 맹점·§5.3 flag gate 자동화. 단 이는 P1 에 *추가만* 하는 비파괴 step 이고 `_evaluate_health` 로직은 불변.
- **주의:** M2 는 발행 파이프라인 인접 코드이므로 루트 평면 .py + 회귀/Codex 검토 게이트를 거친다(golden 불변). EVAL-1 의 E2 시계열 회귀와 자연 결합.

## 7. 주간 스코어카드 출력 포맷 (M1)

```
GRM MON-P2 주간 관측 요약 — 기준일 {D} (윈도우 [D-6, D], lookback {L}일)

A. 운영 run 요약 (ⓒ)
   - 최근 7일 Actions run: 성공 N · 실패 M · warning K
   - 실패/warning Issue 링크 (있으면)
   - WORKFLOW_GAP: {없음 | D-x ~ D-y 사이 run 부재 z일}

B. 소스별 적재 추세 (ⓑ)
   | 소스 | 등급 | 활성 | 최근7일 적재 | 마지막 적재일 | 연속0건 | 판정 |
   (활성 소스만; 비활성은 "(off)"로 별행 표기)

C. 이상 신호
   - [WARN] {소스}: 연속 {일}일 0적재 (등급 임계 {임계} 초과) — run 정상이므로 소스 침묵 의심
   - [INFO] {소스}: 추세 이탈(직전 baseline {g}일 → 현재 {c}일) — 약신호, 모니터
   - [INFO] 관측 부족: {소스 목록} (baseline 산출 불가)
   - 이상 없음이면 "이상 신호 없음"

D. 조치 제안 (해당 시)
   - WARN 항목별 1줄: 수동 확인 포인트(셀렉터·소스 페이지·flag 상태)
```

**알림 규칙:** C 에 WARN 이 1건이라도 있으면 기존 운영 경고 Issue 에 요약 댓글 누적(결정4). INFO/이상없음만이면 스코어카드만 누적(알림 없음 — 과알림 방지).
