## 6. Notion Intake DB 데이터 계약 (행 종류별)

DB `7784c71fb7b343749b2bee5d04db7926` · data source `collection://d5b9634a-2bd7-4036-ba06-e4ad17ede288`. 소스 정본 = `grm_handoff.py` · `delta_bridge.py` · v16 프롬프트 [0단계]·[산출물 예치].

### 6.1 routine-handoff (수집기 → Routine)

- **Title** `OPEN GRM Routine Handoff {run_date}` / 소비 후 `CONSUMED …` / 미소비 봉인 `STALE …` · **Source**=`GRM Handoff` · **Type or Class**=`routine-handoff` · **Status** New(OPEN)→Processed(CONSUMED)/Skipped(STALE).
- 본문 code block = `grm-routine-handoff/v2` JSON: `rows[]`(전 카드 — `page_id`·`source`·`document_id`·`signal_tier`·`card_scaffold`·`prose_input`·`deep_analysis_ready`/`deep_analysis_input.body_full`·`merged_into`…). v1이면 v16 Routine은 **중단**(schema 하드가드).
- 선택 규약(K4-1): Routine은 "최신 `run_date_kst` OPEN 1건"만 소비(날짜 계산 금지). emit 측은 새 OPEN 생성 전 직전 OPEN을 STALE 봉인(항상 OPEN ≤1).
- 마감(PL-10): 발행 후 row Status 갱신(Processed/Skipped/Error) → handoff CONSUMED. 보류분은 handoff 본문에 `DEFERRED N: source::doc_id,…` append(다음 주 PL-10b 재유입 가드의 제외 목록).

### 6.2 web-delta (Routine → 브릿지)

- **Title** `OPEN GRM Web Delta {publish_date}` / 브릿지 소비 후 `CONSUMED …` · **Source**=`GRM Web Delta` · **Type or Class**=`web-delta` · **Status** New→Processed.
- **반드시 DB 행**(부모=data source). 독립 페이지로 만들면 브릿지 쿼리(`web-delta`∧`New`)에 안 잡혀 조용히 유실(2026-07-12 실장애) — v16은 예치 후 재쿼리 자기검증을 강제.
- 본문 **코드블록 1** = 슬롯 델타 envelope `{"cards":{"<card.id>":{title_issue,summary,key_facts[],implication,checks[],quotes_translation[]}},"tldr":[3]}`.
- 본문 **코드블록 2(선택)** = deep 델타 `{"<card.id>":{"deep_analysis":{4섹션},"source_text":"<body_full 원문>"}}` — `source_text` 누락 시 그 카드는 게이트에서 drop. (별도 `OPEN GRM Web Deep Delta {date}`·`web-deep-delta` 페이지도 허용.)
- 멱등: 같은 주 재실행은 기존 OPEN 페이지 본문 **덮어쓰기**(새 페이지 금지). CONSUMED 전환은 **브릿지가 git push 성공 후에만** 수행(Routine이 직접 바꾸지 않음).

### 6.3 상태기계 요약

- 수집 row: `New` →(Routine) `Processed`/`Skipped`/`Error` (보류=미변경+DEFERRED 기재 → 다음 주 정식 재유입).
- handoff: OPEN(New) → CONSUMED(Processed) | STALE(Skipped — 미소비 봉인, 그 미발행 row는 Handoff Ref 해제로 재투입).
- web-delta: OPEN(New) → CONSUMED(Processed) — 브릿지 push 성공 후.

## 9. 파이프라인 단계별 필요성 진단 (2026-07-13 — 전부 ★제안만, 구현 없음)

각 장치에 대해 "왜 존재하나 / 무엇이 깨지면 잡나 / 제거·통합 가능한가"를 평가한다. 판단 근거가 바뀌면(특히 7/20 관찰 후) 이 절을 갱신한다.

### 9-1. 로컬 백업 2겹(09:05 + 14:15) vs GitHub 크론 — 중복 정리 방향

- **왜 생겼나**: 09:05 태스크의 존재 이유였던 3가지 — ①델타 오예치 복구 ②deep 백필 ③크론 미발화 대비 선구동 — 는 각각 PR#236(예치 자기검증), PR#241(deep 클라우드화), 7/13 크론 발화 확증으로 **클라우드가 자체 해결하는 구조가 됐다**. 즉 09:05는 "근본해결 전의 보정층"이었고 근본해결이 전부 배선됐다.
- **판단**: 7/20에 ⓐ클라우드 체인 무개입 완주 ⓑdeep 델타 동봉 첫 실전 통과가 확인되면, 09:05 태스크는 **행동층→관찰층으로 강등**(복구·백필·선구동을 수행하지 말고 이상 시 ★보고만) 후 몇 주 뒤 제거가 합리적. 두 자동화가 같은 일을 하면 경합(예: 로컬 선구동과 크론 브릿지가 같은 델타를 두 번 처리 — 멱등이라 무해하지만 진단 로그가 오염됨)이 생긴다.
- **14:15 ops는 반대로 유일 기능 보유**: watchlist environment 승인 대행은 다른 어떤 자동화도 못 한다(§9-6 참조). 레인 건강 점검·주간 보고도 유일. **걷어낼 층은 09:05, 유지·재활성할 층은 14:15**다(2026-07-13 현재 14:15가 비활성인 것은 역설적 — 재활성 필요★).

### 9-2. Routine(07:30) → 브릿지(09:30) 2시간 간격

- **왜 2시간인가**: Routine 실행 소요(수십 분) + claude.ai 스케줄 지연 흡수. 브릿지가 델타 예치 **전에** 돌면 클린 skip으로 끝나 그 주 이관이 수동 dispatch 의존이 된다.
- **판단**: 간격 **단축은 비권장**(GitHub cron 자체가 +3~4h 지연되는 마당에 상한을 조이면 skip 확률만 커짐). 대신 브릿지 cron을 **하루 2회(예: 09:30·12:30)로 늘리는 것**이 안전한 개선 — 멱등이라 2번째 발화는 무해하고, 1번째가 Routine 지연으로 skip해도 2번째가 잡는다. → 백로그 B2.

### 9-3. watchdog(10:00)의 실효성 — 감시자가 같은 인프라를 공유

- **무엇을 잡나**: 주 위협 모델 = "claude.ai Routine 침묵 미실행". 감시자(GitHub cron)와 감시 대상(claude.ai)이 **다른 플랫폼**이므로 이 모델에는 유효하다.
- **잔여 구멍**: GitHub Actions scheduler 자체가 죽으면 브릿지·watchdog이 함께 침묵(공통 원인 실패) — 이땐 로컬 14:15 ops가 3차 방어선이고, 그마저 데스크톱 OFF면 무경보다. 완전 독립 감시가 필요하면 외부 dead-man's switch(예: healthchecks.io — 브릿지 성공 시 ping, 미수신 시 이메일)가 정석. → 백로그 B3(선택·저우선).
- **판단**: 유지. 제거 대상 아님.

### 9-4. brief-audit(11:00)와 사람 승인 시점의 결합 약점

- **실체**: 감사 대상은 **main에 머지된 최신 `web/data/briefs/brief_web_{date}.json`**이다(워킹트리 직접 읽기·네트워크 0, Notion은 폴백). 즉 11:00에 사람이 아직 승인(머지) 전이면 **지난주 발행본을 재감사**하고, 이번 호는 다음 주 월요일에야 감사된다 — "발행 ~3.5h 후" 주석은 승인이 오전에 끝난다는 낙관 가정이다.
- **판단**: cron 유지보다 **머지 트리거 추가가 정확**하다 — `on: push: paths: web/data/briefs/**`(main)를 추가하면 승인 직후 그 호를 즉시 감사한다(cron은 안전망으로 병존). → 백로그 B4.

### 9-5. Notion 병행 표시(P4 롤백 안전망)·K4 잔여 책임

- **병행 표시**: v16 프롬프트 [산출물] 절에 "(Notion 병행 표시도 동일 JSON에서 파생 — 롤백 안전망, P4 병행기간)" 문구가 잔존하나, `🌐 GRM Weekly Brief` DB는 2026-06-22 이후 레거시(신규 쓰기 중단)로 병행기간은 사실상 종료됐다. 죽은 지시가 프롬프트에 남아 있으면 Routine이 불필요한 행동을 시도할 여지가 있다 — 문구 제거 후보. → 백로그 B5a.
- **K4 잔여(Status 갱신 책임이 LLM에)**: Intake row Status(Processed/Skipped/Error) 갱신·handoff CONSUMED를 여전히 Routine LLM이 수행한다. 브릿지가 web-delta를 consume하는 시점에 **결정론(python)으로 일괄 마감**하도록 옮기면 LLM 책임이 "델타 예치"로만 수렴해 실패 표면이 준다. 단 Skipped/Error 판정 자체는 LLM 판단이므로 완전 이관은 불가 — "델타에 처리 결과표를 동봉하고 브릿지가 반영"하는 절충 설계 검토. → 백로그 B5b(코드 변경·중규모).

### 9-6. 뉴스레터/워치리스트 승인 구조의 비대칭

- **현황**: 뉴스레터=승인 없는 완전 자동(2026-07-05 d92b301에서 environment 게이트 의도적 제거 — 멱등·검증·Admin 중복차단이 경계) vs watchlist=environment 승인 필수(로컬 14:15가 대행 설계).
