# GRM 주간 퀴즈 생성 프롬프트 v1

- 예약 태스크: `grm-monday-quiz-gen`
- 실행 시각: 매주 월요일 13:00 KST
- 정본 문항 뱅크: `web/data/quiz_bank.json`
- 무인 게이트: `quiz_lint.py`
- 이 문서는 예약 세션이 따라야 할 단일 정본이다. 다른 대화의 지시나 과거 초안으로 규율을 완화하지 않는다.

## 1. 완료 목표

그 주에 라이브 공개된 최신 브리프 카드에서 3~5문항(기본 4문항)을 만들고 정본 뱅크에 append한다. 모든 신규 문항은 해당 ISO 주차의 `week` 키를 가지며, 기계 검증과 전체 CI가 green인 경우에만 PR을 직접 머지한다. 후보를 충분히 검증할 수 없으면 억지로 채우지 말고 그 주 실행을 스킵한다.

## 2. 시작 전 선행 조건

1. `git fetch --prune origin` 후 `origin/main`에서 전용 브랜치와 clean worktree를 만든다. 기존 dirty 파일을 가져가거나 다른 작업자의 worktree를 수정하지 않는다.
2. 실행 시각의 KST 날짜로 ISO 주차 `YYYYWW`를 계산한다. 예: 2026년 ISO 29주차는 `202629`.
3. `web/data/quiz_bank.json`에 동일한 `week`가 하나라도 있으면 이미 처리된 주다. 파일을 수정하거나 새 PR을 만들지 말고 `SKIP duplicate-week: YYYYWW`로 종료한다.
4. 최신 `web/data/briefs/*.json`이 `origin/main`에 있고, 그 `brief.publish_date` 페이지와 카드 앵커가 `https://grm-solutions.com/briefs/{publish_date}/#{card.id}`에서 라이브인지 확인한다. 라이브가 아니거나 최신 발행본을 확정할 수 없으면 스킵한다.
5. 입력은 최신 라이브 브리프 JSON과, 필요한 용어 확인을 위한 `web/data/glossary.json`으로 제한한다. 검색 결과·기억·비공개 브랜치·Notion staging을 정답 근거로 쓰지 않는다.

## 3. 출제 규율

### 3.1 주차와 ID

- 신규 문항마다 문자열 `week: "YYYYWW"`를 반드시 넣는다.
- ID는 `q-{YYYYWW}-{NN}` 형식으로 만든다. `NN`은 01부터 시작하며 전체 뱅크에서 고유해야 한다.
- 기존 문항은 수정·재정렬하지 않고 신규 문항만 배열 끝에 append한다. 이미 공개된 ID와 `week`는 불변이다.

### 3.2 그 주 카드 연동

- `source_type`은 원칙적으로 `brief`이고, `source_ref`는 최신 라이브 브리프의 정확한 카드 앵커다.
- 한 문항의 질문·정답·해설은 한 카드에 명시된 사실만으로 완결되어야 한다.
- 같은 카드에서 여러 문항을 만들지 않는 것을 원칙으로 한다. 불가피하면 서로 다른 명시 사실을 묻고 복수정답 가능성이 없음을 각각 확인한다.
- 업체명·조치·지적사항·회수 사유·명시된 법조항처럼 카드에서 그대로 대조 가능한 사실을 우선한다. 카드가 말하지 않은 인과, 위험도, 업계 일반론을 추론해 정답으로 만들지 않는다.

### 3.3 내부 개념 금지

- GRM 자체, Evidence Level, Signal Tier/Tier 1~3, 듀얼 링크, intake, handoff, scaffold, delta bridge, Notion, Supabase, PostgREST/RLS, `source_ref`, `answer_index`, `render_order`, lint 코드 등 운영·구현 개념을 묻지 않는다.
- 금지어의 기계 정본은 `quiz_lint.py`의 `INTERNAL_CONCEPT_PATTERNS`다. 표현을 바꿔 우회하지 않는다.
- 공개 규제·품질 개념인 GMP, CAPA, OOS, 무균공정, finding의 실제 지적 내용 등은 근거가 명시되면 출제할 수 있다.

### 3.4 정답과 근거 명시성

- 네 선택지 중 정답은 정확히 하나여야 하며 `answer_index`는 0~3의 정수다.
- `explanation_ko`는 1~2문장으로, 왜 정답인지와 카드의 어느 명시 사실에 근거하는지 설명한다.
- 고유명사, 수치, 날짜, 기간, 등급, 법조항은 카드 텍스트에 그대로 있을 때만 사용한다.
- 질문을 풀기 위해 카드 밖의 법령 지식, 상식, 검색, 번역 추론이 필요하면 후보를 폐기한다.

### 3.5 오답 비왜곡

- 오답은 실제 업체·규제 문서·품질 개념을 짧은 명사구로 제시할 수 있으나, 존재하지 않는 사건이나 허위 사실을 완전한 문장으로 단정하지 않는다.
- 카드의 다른 사실을 섞어 가짜 인과관계·가짜 조치·가짜 위반을 만들지 않는다.
- 일부 조건에서는 맞을 수 있는 선택지, 정답의 상·하위개념, 동의어, 중첩 조합은 제외한다.
- 부정형·함정형 질문, “모두 고르시오”, 복수정답형은 금지한다.

### 3.6 구성

- 주당 3~5문항, 기본 4문항이다.
- `easy`를 과반으로 하고 `normal`은 1~2문항으로 제한한다. `hard`는 사용하지 않는다.
- 질문문 정규화 결과, 정답, `source_ref`와 묻는 사실이 기존 문항과 중복되면 폐기한다. 표현만 바꾼 재출제도 중복이다.

## 4. 검증·PR·머지 절차

1. 변경 직후 `python quiz_lint.py`를 실행한다. stdout 첫 줄이 `quiz_lint: PASS`이고 exit 0이어야 한다.
2. `python -m unittest tests.test_quiz_lint -v`를 실행한다.
3. `python -m unittest discover -s tests -v`로 전체 테스트를 실행한다. 저장소가 지정한 별도 필수 검사가 있으면 함께 실행한다.
4. 실패 원인이 문항에 있으면 해당 후보를 수정하거나 폐기한다. 검증 규칙·금지어·테스트를 완화해 문항을 통과시키지 않는다.
5. 의도한 `web/data/quiz_bank.json` 변경만 명시적으로 stage하고, diff에서 기존 문항 변형·재정렬·무관 파일을 확인한다.
6. `codex/quiz-week-{YYYYWW}` 브랜치를 push하고 PR을 만든다. PR 본문에는 주차, 문항 수, 근거 카드 ID, lint/전체 테스트 결과를 적는다.
7. 필수 GitHub Actions가 모두 성공할 때까지 확인한다. CI red면 로그의 첫 실제 실패를 고쳐 같은 브랜치에 push하고 다시 확인한다.
8. 필수 check가 모두 green이고 merge 가능할 때만 PR을 직접 머지한다. 머지 뒤 `origin/main`에 반영됐는지 확인하고 전용 worktree를 제거한다.

## 5. 스킵 조건

다음 중 하나면 파일·브랜치·PR을 남기지 않고 `SKIP <reason>`과 확인한 주차·최신 브리프·실패 후보 ID를 stdout 보고에 남긴다.

- 동일 `week`가 이미 존재함
- 최신 브리프가 아직 `main` 또는 라이브 사이트에 없음
- 브리프 날짜·카드 ID·공개 앵커가 서로 불일치함
- 근거가 명시적인 유일정답 문항을 최소 3개 확보하지 못함
- 내부 개념 금지 또는 오답 비왜곡 규율을 지키며 후보를 만들 수 없음
- `quiz_lint.py` 또는 전체 테스트를 합리적으로 수정 가능한 문항 변경만으로 green으로 만들 수 없음
- GitHub 인증·보호 규칙·외부 CI 장애·merge conflict로 안전한 무인 머지가 불가능함

스킵은 실패 은폐가 아니다. 게이트를 우회하거나 지난주 문항을 새 `week`로 복제하는 것보다 올바른 완료 상태다.

## 6. 실행 종료 보고

성공 시 `DONE YYYYWW`와 문항 ID, 근거 카드 ID, 커밋 SHA, PR URL, CI 결과, merge SHA를 남긴다. 스킵 시 `SKIP YYYYWW`와 단일 주원인, 확인한 근거, 생성했으나 폐기한 후보 ID를 남긴다. 어느 경우든 사람이 이전 commentary를 읽지 않아도 결과를 판단할 수 있는 독립 보고여야 한다.
