# GRM 031 인기 카드 RPC 적용·검증 계획

> 상태: **적용 전 초안** · 작성일: 2026-07-18
> 대상: `web/migrations/031_reactions_weekly_top.sql`
> 금지: 이 문서 작성 단계에서는 Supabase 프로젝트에 적용하거나 라이브 데이터를 변경하지 않는다.

## 1. 적용 대상과 고정 계약

RPC 시그니처는 `public.reactions_weekly_top(p_limit integer default 3)` 하나다. `p_limit`은 `NULL`이면 3이며, 서버에서 1~5로 clamp한다. 별도 시작일·종료일·정렬식은 받지 않는다.

반환 필드는 아래 둘이 전부다.

| 필드 | PostgreSQL 형식 | 의미 |
|---|---|---|
| `card_id` | `text` | 반응 테이블의 불투명 카드 앵커 |
| `distinct_user_count` | `bigint` | 이번 KST 주간에 생성되어 현재도 활성인 heart/scrap 반응의 카드별 고유 사용자 수 |

`user_id`, 이메일, 개별 `created_at`, `kind`, `hearts`, `scraps`, 제목, 요약, 기관, 원문 URL, `raw_json`은 반환하지 않는다. `hearts`와 `scraps`는 동률 정렬에만 내부 사용한다. 후속 클라이언트는 반드시 RPC의 `card_id`와 커밋된 `search-index.json`의 앵커를 교차하고, 인덱스에 없는 ID를 버린다. 제목·링크는 정적 인덱스에서만 얻는다.

주간 창은 `Asia/Seoul` 기준 이번 주 월요일 00:00 이상, 함수 조회 시점 이하이다. 현재 `public.reaction`에 남은 행만 읽으므로 삭제된 반응은 제외된다. 한 사용자가 같은 카드에 heart와 scrap을 모두 남겨도 점수는 1이다. 정렬은 `distinct_user_count DESC`, 내부 `scraps DESC`, 내부 `hearts DESC`, `card_id ASC`다.

함수는 본인 행만 보이는 `reaction` RLS를 넘어 공개 집계를 계산해야 하므로 `SECURITY DEFINER`다. `search_path`를 고정하고 원천 관계를 `public.reaction`으로 한정한다. 함수 생성 직후 `PUBLIC`, `anon`, `authenticated`, `service_role`의 기본 실행권을 회수한 뒤 `anon`, `authenticated`에만 `EXECUTE`를 다시 부여한다.

031은 함수·주석·함수 권한만 추가한다. 기존 테이블·RLS·정책·트리거·뷰를 바꾸지 않고, Edge Function·이벤트 테이블·인덱스를 만들지 않는다. 현재 라이브 반응량과 실행계획이 미확인이므로 인덱스는 실측 없이 선제 추가하지 않는다.

## 2. 적용 전 게이트

1. 대상 커밋의 변경 파일이 아래 둘뿐인지 확인한다.

   ```text
   web/migrations/031_reactions_weekly_top.sql
   docs/specs/GRM_031_적용검증계획_2026-07-18.md
   ```

2. `git diff --check`가 통과하고, 031의 DDL 표면이 함수 1개·함수 주석 1개·함수 권한 회수/부여뿐인지 검토한다.
3. 대상 환경에서 아래 사전 조회가 `NULL`인지 확인한다. 기존 함수가 있으면 이름 충돌이므로 적용하지 말고 정의·소유자·호출자를 먼저 조사한다.

   ```sql
   select to_regprocedure('public.reactions_weekly_top(integer)');
   ```

4. 운영 DB가 아닌 폐기 가능한 로컬 Supabase 또는 비프로덕션 브랜치에서 먼저 검증한다. 운영 데이터베이스에서 빈 데이터 시험을 위해 `reaction`을 삭제해서는 안 된다.
5. 비프로덕션 검증용 사용자는 Supabase Auth의 정상 경로로 최소 3개 준비한다. `public.reaction.user_id`의 FK를 우회하거나 운영 사용자 ID를 복사하지 않는다.

## 3. SQL dry-run과 실행계획 실측

### 3.1 트랜잭션 dry-run

폐기 가능한 DB의 `psql`에서 031을 한 트랜잭션 안에 로드하고 마지막에 반드시 롤백한다. 파일 경로는 실행 환경에 맞게 바꾼다.

```sql
begin;
\i 'web/migrations/031_reactions_weekly_top.sql'

select * from public.reactions_weekly_top();
select * from public.reactions_weekly_top(null);
select * from public.reactions_weekly_top(0);
select * from public.reactions_weekly_top(999);

rollback;
```

판정 기준은 구문·권한 DDL 오류가 없고, 기본/`NULL`은 최대 3행, 0 이하는 최대 1행, 5 초과는 최대 5행을 반환하는 것이다. 데이터가 없으면 네 호출 모두 오류나 `NULL` 행 없이 0행이어야 한다.

### 3.2 `EXPLAIN (ANALYZE, BUFFERS)`

비프로덕션 데이터 스냅샷에서 콜드 1회와 웜 5회를 기록한다.

```sql
explain (analyze, buffers, verbose, settings)
select * from public.reactions_weekly_top(3);
```

다음 항목을 검증 기록에 남긴다.

- 실제 반환 행이 3 이하이고 함수 오류가 없는지
- planning/execution time, shared hit/read, temp read/written, 정렬 방식과 메모리
- `public.reaction` 예상행과 실제행의 차이, 주간 조건에서 제거된 행 수
- 반복 호출에서 계획과 결과 순서가 안정적인지

temp spill, 비정상적으로 큰 예상/실제 행 차이, 또는 허용하기 어려운 지연이 확인되면 운영 적용을 중단한다. 인덱스가 필요하다는 근거가 생겨도 031에 추정으로 끼워 넣지 말고, 라이브 반응량·쓰기 비용·후보 인덱스 계획을 별도 마이그레이션으로 검토한다.

### 3.3 KST 경계와 distinct-user fixtures

비프로덕션 Auth 사용자 UUID를 사용해 다음 상태를 만든다.

- 주 시작 1마이크로초 전 반응: 제외
- 주 시작과 정확히 같은 시각의 활성 반응: 포함
- 조회 시점 이후 시각의 반응: 제외
- 같은 사용자·같은 카드의 heart+scrap: `distinct_user_count` 1
- 서로 다른 사용자 2명의 반응: `distinct_user_count` 2
- 고유 사용자 수 동률: scraps, hearts, `card_id` 순으로 결정론 정렬
- 삭제 후 재호출: 삭제된 상태는 제외

주 시작값은 함수와 같은 식으로 계산한다.

```sql
select date_trunc('week', now() at time zone 'Asia/Seoul')
         at time zone 'Asia/Seoul' as kst_week_start;
```

fixture `card_id`는 `__rpc031_test__` 접두사를 쓰고 시험 직후 모두 삭제한다. 비프로덕션 브랜치라도 실제 카드 ID를 fixture로 재사용하지 않는다.

### 3.4 빈 데이터 주간

폐기 가능한 로컬 DB 또는 전용 비프로덕션 브랜치에서만 아래를 수행한다. 트리거가 집계표도 갱신하므로 운영 DB에서는 실행 금지다.

```sql
begin;
delete from public.reaction;

select count(*) as returned_rows
from public.reactions_weekly_top(3);

rollback;
```

`returned_rows = 0`이어야 한다. 비프로덕션 PostgREST에서도 같은 상태의 응답은 HTTP 200과 JSON `[]`여야 하며 `null`, `{}`, 404, 5xx가 아니어야 한다.

## 4. anon 키 PostgREST 실측

SQL dry-run 통과 뒤 031을 **비프로덕션 Supabase 브랜치에만** 적용해 스키마 캐시를 갱신하고 측정한다. service-role 키를 브라우저·명령행 인자·로그에 넣지 않는다. 아래 변수에는 비프로덕션 URL과 anon key만 둔다.

```bash
export SUPABASE_URL='https://<non-production-ref>.supabase.co'
export SUPABASE_ANON_KEY='<non-production-anon-key>'

curl --fail-with-body --silent --show-error \
  "$SUPABASE_URL/rest/v1/rpc/reactions_weekly_top" \
  -H "apikey: $SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_ANON_KEY" \
  -H 'Content-Type: application/json' \
  --data '{}'
```

동일 호출을 `{"p_limit":null}`, `{"p_limit":0}`, `{"p_limit":1}`, `{"p_limit":5}`, `{"p_limit":999}`로 반복한다. 각 응답은 HTTP 200이어야 하고 행 수는 각각 최대 3, 3, 1, 1, 5, 5다.

응답 allowlist는 기계적으로 고정한다.

```bash
curl --fail-with-body --silent --show-error \
  "$SUPABASE_URL/rest/v1/rpc/reactions_weekly_top" \
  -H "apikey: $SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_ANON_KEY" \
  -H 'Content-Type: application/json' \
  --data '{"p_limit":5}' \
| jq -e '
    type == "array" and
    all(.[];
      (keys | sort) == ["card_id", "distinct_user_count"] and
      (.card_id | type) == "string" and
      (.distinct_user_count | type) == "number" and
      .distinct_user_count >= 1
    )'
```

정확히 두 키만 허용하므로 사용자 ID·시각·kind별 수·콘텐츠·URL 필드 부재도 함께 검증된다. fixture의 예상 순서/카운트와 응답을 대조하고, 빈 주간은 `[]`인지 재확인한다.

함수 속성과 권한은 SQL로 별도 확인한다.

```sql
select
  p.prosecdef as security_definer,
  p.provolatile = 's' as is_stable,
  p.proconfig,
  pg_get_function_result(p.oid) as result_type,
  has_function_privilege('anon', p.oid, 'execute') as anon_execute,
  has_function_privilege('authenticated', p.oid, 'execute') as authenticated_execute,
  has_function_privilege('service_role', p.oid, 'execute') as service_role_execute,
  not exists (
    select 1
    from aclexplode(p.proacl) a
    where a.grantee = 0
      and a.privilege_type = 'EXECUTE'
  ) as public_execute_revoked
from pg_proc p
where p.oid = 'public.reactions_weekly_top(integer)'::regprocedure;
```

기대값은 `security_definer=true`, `is_stable=true`, `proconfig`에 `search_path=public`, 반환 형식이 `TABLE(card_id text, distinct_user_count bigint)`, anon/authenticated만 `true`, service_role은 `false`, `public_execute_revoked=true`다.

## 5. 운영 적용과 즉시 확인

컨트롤타워는 위 게이트가 모두 통과한 뒤에만 승인된 031 파일을 운영 Supabase에 1회 적용한다. 적용 직후 다음을 수행한다.

1. anon PostgREST로 `{}`와 `{"p_limit":5}`를 호출해 HTTP 200·필드 allowlist·상한을 재확인한다.
2. 같은 시각 SQL의 독립 `count(distinct user_id)` 집계와 RPC 순서/카운트를 대조한다. SQL 결과나 로그에 사용자별 행을 복사하지 않는다.
3. 실행계획과 응답시간을 기록하되, 현재 라이브 반응량을 최초 기준선으로 남긴다.
4. 검색 인덱스와 교집합을 계산해 미등록 `card_id` 수만 기록한다. 미등록 ID의 문자열을 공개 로그나 UI에 노출하지 않는다.
5. 이상이 있으면 웹 통합을 진행하지 않고 아래 롤백을 실행한다.

## 6. 롤백

031은 신규 함수이므로 롤백은 함수 삭제다. 함수 주석과 권한도 함께 제거된다.

```sql
begin;

drop function if exists public.reactions_weekly_top(integer);
notify pgrst, 'reload schema';

commit;
```

롤백 뒤 `to_regprocedure('public.reactions_weekly_top(integer)') is null`을 확인하고, anon RPC가 더는 성공하지 않는지 확인한다. 기존 `reaction`, `reaction_count`, `heart_counts`, RLS, 트리거 데이터는 변경하지 않는다.

## 7. 후속 웹 통합 계약 테스트

후속 웹 PR은 최소한 아래를 자동 테스트한다.

- RPC 호출 인자는 `p_limit` 하나뿐이며 기본 요청은 빈 객체다.
- 응답 객체의 키 집합은 정확히 `card_id`, `distinct_user_count`다.
- `distinct_user_count`를 숫자로 처리하고, 사용자 ID·개별 시각·kind별 수를 기대하거나 파생하지 않는다.
- RPC `card_id`와 커밋된 `search-index.json`을 교차하며 미등록 ID는 렌더하지 않는다.
- 제목·기관·상세 링크는 정적 인덱스에서만 읽고 RPC 값으로 덮어쓰지 않는다.
- 알려진 카드가 부족하거나 제품 임계값 미만이면 인기·순위 표현 없이 결정론적 `이번 주 새 카드` 폴백을 쓴다. 0점 카드를 랭킹 채움용으로 섞지 않는다.
- 네트워크 오류, 401/403/404/5xx, timeout, 잘못된 JSON, 빈 배열은 정적 폴백으로 수렴하며 페이지 렌더를 막지 않는다.
- 순위는 RPC 배열 순서를 보존하고 클라이언트에서 임의 재가중하지 않는다.
- 로그인 전 anon과 로그인 후 authenticated 호출의 공개 결과가 동일하다.
- 브라우저 번들·HTML·로그에 service-role 키가 없고 anon key만 현행 방식으로 주입된다.
- 하트/스크랩 toggle 후 활성 상태와 주간 의미가 일치하고, 삭제된 반응은 다음 조회에서 점수에 남지 않는다.
- KST 월요일 경계, `p_limit` NULL/하한/상한, 동일 사용자 이중 kind, 동률 `card_id` 정렬을 회귀 fixture로 고정한다.
- RPC 장애가 기존 하트 수, 로그인, 스크랩, 브리프·Findings 정적 페이지에 영향을 주지 않는다.
- Edge Function·조회 이벤트 테이블·외부 분석 서비스가 추가되지 않았는지 변경 파일과 네트워크 호출을 검사한다.

## 8. 적용 전까지 남는 미확인 사항

- 라이브 `reaction` 전체 행 수와 이번 KST 주간 활성 행 수
- 카드별 분포, 고유 반응 사용자 수, 미등록/임의 `card_id` 비율
- 실제 실행계획, temp spill 여부, 콜드·웜 지연과 PostgREST 왕복시간
- 인기 섹션 표시 임계값을 정할 수 있는 라이브 분포
- 라이브 Auth의 이메일 확인 강제 여부와 다계정 어뷰즈 수준

이 값들은 컨트롤타워의 anon read-only 실측과 비프로덕션/운영 적용 게이트에서 확인한다. 확인 전에는 인덱스나 제품 임계값을 확정하지 않는다.
