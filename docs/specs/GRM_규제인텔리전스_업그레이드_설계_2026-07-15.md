## 4. 상세 설계 A — 유사 문구 검색(S1) + 유사 사례(S2)

### 4.0 공통 안전 계약 (v1.1 정정 — 불가침)

1. **공개 술어 = 현행 010 정책과 동일**: `(finding_text_ko <> '' or finding_language = 'KO') and scope_status = 'ok'` — S1 결과·S2 기준 finding·S2 후보 **셋 모두**에 적용. 006만 복제했던 v1.0 술어는 폐기.
2. **문서 정체성 = `raw_signal_id`**: 같은 문서 제외·문서 수 집계·딥링크 전부 raw_signal_id 기준(웹 groupByDocument와 동일 키).
3. 반환 필드는 row 조회로 이미 공개되는 것만 + score. 벡터·raw 미반환.

### 4.1 임베딩 방식 (확정)

**(c) 빌드타임 로컬 multilingual-e5-small 확정.** 정정 사항:

- 모델 파일 **fp32 ≈ 471MB**(ONNX int8 대안 ≈ 112MB — 구현 시 선택·실측). `actions/cache` 캐시. 저장소 PUBLIC(실측 확인)이라 표준 러너 과금 없음. **CI 소요시간은 초기 전량 임베딩 dry-run에서 실측 후 이 문서에 기재**(v1.0의 "+2분"은 가정이었음).
- **벡터 공간 비호환 정정**: S3에서 쿼리 공급자를 OpenAI/Gemini로 바꾸면 **코퍼스 전량 재임베딩 필요**(0.4M 토큰 — CI 1회 실행 규모라 치명적이진 않으나 "그냥 붙이면 된다"는 틀린 서술이었음). E5를 유지한 채 쿼리 임베딩만 셀프호스팅하는 경로만 재임베딩 면제.
- **prefix**: 아이템-투-아이템은 대칭 유사도 → E5 모델 카드 권고대로 **양쪽 `query:`** prefix(비대칭 `passage:` 아님). 클러스터링(§5 패턴카드)도 동일.

### 4.2 S1 — 유사 문구 검색 (렉시컬, 조건부 착수)

- **명칭**: UI·문서 모두 "유사 문구 검색" (의미검색 아님을 정직 표기).
- **검색 대상 텍스트(v1.1.1 정정)**: `finding_text_ko` 단독이 아니라 **`coalesce(nullif(finding_text_ko,''), finding_text)`** — 원문이 한국어인 행(finding_language='KO')은 finding_text_ko가 비어 있어도 공개 게이트를 통과하므로, ko 단독 검색이면 이 행들이 검색에서 누락된다. 이하 이 표현식을 `search_text`로 표기.
- **RPC 구조 정정 — 후보 검색 → 재랭킹 2단**: `similarity()` 단일 ORDER BY는 인덱스를 못 탄다. ① 후보: `search_text % p_query`(trgm expression GIN 인덱스 가용) OR `websearch_to_tsquery` 매치로 상위 ~200 후보 추출 → ② 재랭킹: `0.6*similarity + 0.4*ts_rank` (가중치는 표본 쿼리로 튜닝).
- **인덱스 2개 명시(둘 다 expression GIN, 동일 표현식과 byte 일치 필수 — 불일치 시 인덱스 미사용)**:
  ```sql
  create index if not exists idx_findings_search_trgm
    on public.findings using gin
    ((coalesce(nullif(finding_text_ko, ''), finding_text)) gin_trgm_ops);
  create index if not exists idx_findings_search_fts
    on public.findings using gin
    (to_tsvector('simple', coalesce(nullif(finding_text_ko, ''), finding_text)));
  ```
- **배포 전 `EXPLAIN ANALYZE` 필수** — 인덱스 사용·지연 실측을 PR 본문에 첨부.
- 입력 가드(길이 2~500·limit≤50 클램프)·security definer + `set search_path` 관례·revoke/grant 관례는 v1.0 유지.
- **중복 붕괴**: 결과에서 동일 finding_text는 1개로 접고 "동일 문구 N개 문서" 배지(59.4% 중복 실측의 완화책 — 상투문구 도배 차단).
- 마이그레이션 `018_findings_similar_lexical.sql`. 롤백: drop function/index + JS 블록 제거.

### 4.3 S2 — 빌드타임 임베딩 "유사 사례" (평가셋 게이트 통과 시에만 공개)

**공개 게이트(신설·차단선)**: 규제 사례 **평가셋 30~50개**(카테고리·상황 다양화, 사람 작성)로 **S1 대비 Precision@5·nDCG@10 개선을 입증한 경우에만** 웹 공개. 미달 시 S2는 내부 자산(§5 클러스터링용)으로만 유지.

**임베딩 입력 A/B 비교(평가셋으로 결정)**:
- A안 = 현행 `finding_text`(deficiency만) — 중복·상투문구 리스크 그대로.
- B안 = `deficiency + detail` 재조합 — detail은 `raw_signals.raw_json`의 observation에 존재(추출기가 버릴 뿐). CI embed job이 service_role로 raw를 읽어 재조합. **중복 59.4%를 구체 사실관계로 분화시키는 정공법** — B안 우세 예상이나 평가셋으로 입증.

**스키마·전환 규율 (v1.1.1 정정 — finding_id 단일 PK로는 구·신 벡터 병렬 적재 불가)**:
- `finding_embeddings(embedding_version int not null, finding_id text not null fk, embedding halfvec(384), model text, embed_input text('A'|'B'), text_sha256, embedded_at, **primary key (embedding_version, finding_id)**)` + RLS 전면차단(노출 표면=RPC 1개).
- **임베딩 버전 원자 전환**: 1행 설정 테이블 `embedding_config(active_version int, ...)`. 신버전은 **구버전 서빙 유지 중 병렬 전량 적재**(복합 PK라 공존 가능) → 적재 완결 검증(건수=공개분 전량·차원 sanity) 후 `active_version` 1행 UPDATE로 원자 전환 → 구버전 행 지연 삭제. RPC는 `where embedding_version = (select active_version from embedding_config)`만 조회(혼합 벡터공간 서빙 구조적 차단). 버전별 model/embed_input 매핑은 config 또는 버전 레지스트리 행에 기록.
- **인덱스**: 초기 **없음(exact cosine)** — 8.5k 규모 순차 스캔 지연 실측 후, 필요 시에만 HNSW 추가(근사+필터 후적용의 결과 부족 리스크 문서화 포함).

**RPC `findings_similar_by_id`**: 기준 finding·후보 모두 §4.0 술어 적용, **같은 raw_signal_id 제외**, 동일 텍스트 붕괴, top-K 반환.

**CI `grm-findings-embed.yml`**: cron 매일(intake 후). 플래그 `ENABLE_FINDINGS_EMBED` 기본 off → dry-run(건수·차원·소요시간 출력) → on. 신규 시크릿 0(기존 SUPABASE_* 재사용).

**롤백**: 플래그 off → 웹 S1 폴백 → drop table·워크플로 삭제. findings 무변경.

### 4.4 S3 — 보류 (재논의 조건 v1.1 정정)

재논의 트리거(S1 사용량 대비 저품질 신호)는 유지하되, 전환 비용 서술 정정: 공급자 교체 = **쿼리 공급자 + 코퍼스 재임베딩 세트**(0.4M 토큰·CI 1회 규모). E5 유지+쿼리 셀프호스팅 경로만 재임베딩 면제.

## 5. 상세 설계 B — 패턴카드 내부 실험 (체크리스트 대체·축소판)

**월간 자동 생성·공개 체크리스트는 불승인.** 승인된 범위:

- **범위**: `aseptic_sterility_assurance` 1개 카테고리(공개분 2,407건 — 표본 최대)만. 모달리티 세그먼트는 데이터 부재(99.6% 공백)로 **폐기**.
- **산출물**: 패턴카드 **5~8개, 1회성 내부 문서**(웹 미공개 — `archive/` 보관·git 미커밋 관례). 카드 = 반복 지적 패턴 서술 + 점검 질문 1개 + 근거 사례 링크.
- **근거 강화 조건(각 카드)**: 서로 다른 **문서(raw_signal_id) 3건 이상 + 업체(firm_key) 2곳 이상 + Evidence A만**. 빈도 표시는 Observation 행 수가 아니라 **distinct 문서·업체 수**(중복 59.4% 왜곡 차단). 날짜는 **"FDA 공개일 기준"** 명기.
- **생성 절차**: S2 임베딩(내부 자산) `query:` prefix 클러스터링 → 예약 세션 1회 생성 → grounding lint(실존·카테고리 일치·비창작 D2류·동어반복 차단 — v1.0 §5.4의 5종 게이트를 lint 스크립트로 유지) → **사람 검토 필수**.
- **승격 조건**: 3부서 파일럿에서 행동가능성·근거열람률·시간절감 신호 확인 후에만 공개판·월간 갱신 재승인 논의.
