## 2. 조사 A — 현행 반응 인프라 역공학

### 2.1 브라우저 호출 경로

반응 기능은 `SUPABASE_URL`과 `SUPABASE_ANON_KEY`가 모두 있을 때만 렌더러가 켠다. service-role 키는 클라이언트에 배선하지 않는다고 코드 주석이 명시한다(`web/render.py:819-824`, `web/render.py:945-951`). 활성 시 `base.html`이 공개 URL·anon key·정적 검색 인덱스 경로를 주입하고 supabase-js와 `reactions.js`를 로드한다(`web/templates/base.html:285-287`, `web/templates/base.html:378-381`).

| 목적 | 현행 호출 | 반환/쓰기 | 근거 |
|---|---|---|---|
| 공개 하트 수 | `heart_counts` view `SELECT ... IN(card_id)` | `card_id`, `hearts` | `web/assets/reactions.js:385-397` |
| 로그인 사용자의 상태 | `reaction` table `SELECT` | 본인의 `card_id`, `kind`만 | `web/assets/reactions.js:399-415`; RLS는 `web/migrations/001_reaction.sql:15-21` |
| 하트·스크랩 토글 | `reaction` table `INSERT` 또는 `DELETE` | `{user_id, card_id, kind}` | `web/assets/reactions.js:425-448` |
| 내 스크랩 | `reaction` table `SELECT kind='scrap'` + 정적 인덱스 fetch | `card_id`, `created_at`; 제목·링크는 로컬 인덱스 | `web/assets/reactions.js:469-500` |
| RPC | **호출 없음** | 해당 없음 | 2026-07-18 `.rpc(` 0건 실측 |
| Edge Function | **호출 없음** | 해당 없음 | 2026-07-18 `functions/v1` 0건 실측 |

즉, `reactions.js`가 호출하는 Supabase 데이터 표면은 `reaction` table과 `heart_counts` view뿐이다. Auth 메서드는 로그인 세션을 만들고 갱신하지만 인기 집계 API는 아니다(`web/assets/reactions.js:15-25`, `web/assets/reactions.js:611-619`).

### 2.2 저장 스키마와 익명성 구조

`reaction`은 `(user_id, card_id, kind)`를 복합 기본키로 갖는다. `kind`는 `heart|scrap`, `created_at`은 생성 시각이며 한 계정이 한 카드에 같은 종류 반응을 동시에 두 개 보유할 수 없다(`web/migrations/001_reaction.sql:5-11`). 본인 행만 SELECT/INSERT/DELETE하도록 RLS가 강제된다(`web/migrations/001_reaction.sql:13-21`).

공개 집계는 trigger가 유지하는 `reaction_count(card_id, kind, total, updated_at)`이다. anon/authenticated에는 이 집계표 SELECT만 열리고(`web/migrations/001_reaction.sql:31-51`), insert/delete 때 총계가 1씩 증감한다(`web/migrations/001_reaction.sql:61-107`). `heart_counts`는 그중 하트 양수 행만 투영한 security-invoker view다(`web/migrations/001_reaction.sql:109-116`).

익명성은 다음처럼 구분해야 한다.

- **공개 사용자에게는 익명:** 공개 표면은 카드별 수뿐이고 `user_id`를 반환하지 않는다. 개별 `reaction` 행은 본인 외에는 RLS로 보이지 않는다(`web/migrations/001_reaction.sql:15-24`).
- **서비스에 대해서는 가명성:** DB에는 Supabase `auth.users(id)`를 참조하는 `user_id`가 저장되므로 운영 백엔드까지 신원을 모르는 완전 익명 구조는 아니다(`web/migrations/001_reaction.sql:2-10`).
- **콘텐츠 최소화:** 반응 DB에는 제목·원문 URL 대신 불투명 웹 앵커만 저장한다(`web/migrations/001_reaction.sql:1-3`). 내 스크랩 화면도 Supabase의 `card_id`를 정적 `search-index.json`과 로컬에서 결합한다(`web/assets/reactions.js:469-486`). 인덱스의 제목·기관·상세 링크는 커밋 브리프에서 결정론적으로 파생된다(`web/render.py:638-658`, `web/render.py:661-691`, `web/render.py:1161-1164`).
- **운영자는 별도 경로:** `admin-supabase` Edge Function은 service role로 `reaction` 최대 5,000행을 읽어 누적 Top 20을 계산한다(`supabase/functions/admin-supabase/index.ts:83-106`). 그러나 모든 요청에 유효 세션과 `admin_user` 행을 요구하고(`supabase/functions/_shared/admin.ts:52-81`), `action=reactions|overview`도 그 게이트 뒤에 있다(`supabase/functions/admin-supabase/index.ts:172-220`). 공개 랭킹 재사용 경로가 아니다.

### 2.3 현행 집계로 가능한 것과 불가능한 것

| 질문 | 기존 집계만으로 가능? | 판정 근거 |
|---|---|---|
| 현재 페이지 카드의 누적 하트 표시 | 가능 | `heart_counts`를 현재 DOM의 ID 목록으로 제한해 읽는다(`web/assets/reactions.js:385-394`). |
| 카드별 현재 하트+스크랩 누적 순위 | 가능 | 공개 `reaction_count`에 두 kind의 현재 total이 있다(`web/migrations/001_reaction.sql:31-51`). 단, 현행 JS는 이를 랭킹에 사용하지 않는다. |
| 이번 주에 생긴 반응 Top N | 불가능 | `reaction_count`에는 시간 버킷이 없고 `updated_at`은 마지막 증감 시각일 뿐이다(`web/migrations/001_reaction.sql:31-36`, `web/migrations/001_reaction.sql:69-85`). 원천 `reaction.created_at`은 공개 조회가 본인 행으로 제한된다(`web/migrations/001_reaction.sql:15-21`). |
| 조회수 Top N | 불가능 | 조회 이벤트 저장·분석 코드가 0건이다(§1 저장소 검색 실측). |
| 기존 Admin Top 20을 공개 재사용 | 불가 | 누적·최대 5,000행 표본이며(`supabase/functions/admin-supabase/index.ts:83-105`) admin 인증 전용이다(`supabase/functions/_shared/admin.ts:52-81`). |

RPC를 만들지 않는 최소 대안은 최신 브리프 카드 ID만 `reaction_count`에서 읽어 **누적 현재 반응**으로 정렬하는 것이다. 이때 문구는 `이번 주`가 아니라 `이번 호 독자 반응` 또는 `독자 반응이 많은 카드`여야 한다. 웹 앵커는 `document_id`를 사용하므로(`card_scaffold.py:308-310`, `web/render.py:153-160`) 같은 문서가 재등장하면 과거 반응도 이어질 수 있기 때문이다. 현재 커밋 데이터에서는 ID 중복이 0건이지만 이는 제약이 아니라 실측 스냅샷이다(§1).

## 3. 인기 카드 v1 권장 설계

### 3.1 의미 계약

권장 제목은 **`이번 주 반응이 모인 카드`**, 보조 설명은 **`하트·스크랩 기준 · 조회수 아님`**이다. `가장 많이 본`은 조회 계측을 별도로 도입하기 전까지 사용하지 않는다.

주간은 `Asia/Seoul` 기준 월요일 00:00부터 현재까지로 고정한다. 후보 반응은 `reaction.created_at`이 그 구간에 들어오고 조회 시점에도 활성인 행이다. 이는 클릭 이벤트 수가 아니라 **이번 주에 새로 만들어져 아직 유지 중인 반응 상태**다. delete된 반응은 세지 않으며, 껐다 다시 켜면 새 `created_at`으로 이번 주 상태 1건이 된다. 이 의미는 현재 toggle이 행을 insert/delete하는 구조와 정확히 맞는다(`web/assets/reactions.js:425-448`, `web/migrations/001_reaction.sql:5-11`).

기본 점수는 카드별 `count(distinct user_id)`로 한다. 하트와 스크랩을 둘 다 누른 한 계정도 인기 점수에는 1명으로만 반영하고, `hearts`·`scraps`는 설명용 보조 카운트로 반환한다. 동률은 `reactors DESC → scraps DESC → hearts DESC → card_id ASC`로 고정한다. 가중치 점수는 대중에게 설명하기 어렵고 어뷰즈 표면을 늘리므로 v1에서 쓰지 않는다.

### 3.2 신규 공개 RPC 안전 계약

후속 구현 시 RPC는 다음 계약을 지켜야 한다.

```json
[
  {"card_id":"opaque-document-anchor","reactors":7,"hearts":6,"scraps":4}
]
```

- 입력은 `p_limit` 하나만 허용하고 기본 3, 범위 1~5로 clamp한다. 임의 시작일·종료일·정렬식은 받지 않고 서버가 KST 주간을 계산한다.
- 반환 allowlist는 `card_id`, `reactors`, `hearts`, `scraps`뿐이다. `user_id`, 이메일, 개별 `created_at`, 제목, 요약, 원문 URL, `raw_json`은 반환하지 않는다.
- `security definer`가 필요한 이유는 원천 `reaction` RLS가 본인 행만 허용하기 때문이다(`web/migrations/001_reaction.sql:15-21`). 함수는 고정 `search_path`, fully-qualified relation, 명시적 권한 회수 후 anon/authenticated execute만 허용해야 한다. 기존 private trigger가 공개 실행권을 전부 회수하는 관례도 확인된다(`web/migrations/001_reaction.sql:61-66`, `web/migrations/001_reaction.sql:94-96`).
- 별도 Edge Function·새 이벤트 테이블·외부 분석기는 두지 않는다. 기존 상태 행을 요청 시 집계하므로 추가 사용자 데이터와 유료 서비스가 0이다. 현재 웹 배포도 GitHub Actions+Cloudflare 무료 구조를 명시한다(`.github/workflows/grm-web-deploy.yml:7-18`).
- RPC의 `card_id`는 반드시 커밋된 `search-index.json`의 앵커와 교집합을 취한다. 인덱스에 없는 ID는 표시하지 않는다. 제목·기관·링크는 RPC가 아니라 정적 인덱스에서만 얻는다(`web/render.py:638-658`, `web/assets/reactions.js:469-486`).
- 실제 마이그레이션 전 anon PostgREST 계약 테스트, 원문/사용자 필드 부재 테스트, KST 주 경계 테스트, 반복 toggle 테스트, `EXPLAIN (ANALYZE, BUFFERS)`를 수행한다. 현재 반응량과 실행계획은 `미확인`이므로 성능 인덱스를 선제 추정해 추가하지 않는다.

신규 RPC 이름은 예시일 뿐이며 기존 migration 명명 규칙에 맞춰 후속 작업에서 확정한다. 이 조사에서는 SQL 파일을 만들지 않는다.

### 3.3 어뷰즈 내성

- 복합 기본키가 계정당 카드·종류별 최대 1행을 강제한다(`web/migrations/001_reaction.sql:5-11`). RPC 점수를 distinct user로 계산하면 하트+스크랩 이중 반영도 막는다.
- 반복 클릭은 insert/delete toggle이므로 최종 활성 상태가 최대 1점이다(`web/assets/reactions.js:431-448`). 클릭 횟수를 이벤트로 저장하지 않아 매크로 반복 자체가 점수를 누적하지 못한다.
- 임의 `card_id` 문자열 insert를 막는 FK/allowlist는 현행 스키마에 없다(`web/migrations/001_reaction.sql:5-10`). 따라서 응답 후 정적 인덱스에 없는 ID를 버리는 방어가 필수다. 알려진 실제 ID를 겨냥한 조작은 계정당 1점 제한만 적용된다.
- 다계정 생성 농장은 DB 기본키로 막을 수 없다. 클라이언트에는 가입 확인 코드 흐름이 구현돼 있지만(`web/assets/reactions.js:126-138`, `web/assets/reactions.js:207-220`), 라이브 Auth의 이메일 확인 강제 설정은 `미확인`이다. v1은 작은 숫자를 과도하게 강조하지 않고, 이상 급증은 Admin 누적표로 운영 점검한다.
- 조회수는 로그인 없이 쉽게 증폭되는 새 이벤트 표면이므로 v1에서 수집하지 않는다. 나중에 도입하려면 봇·중복 세션·보존기간·동의 정책을 별도 설계한다.

### 3.4 노출 위치와 표현

랜딩의 히어로가 끝나는 지점과 `Why GRM` 사이가 v1 위치다. 현재 템플릿에서 히어로는 54행에 끝나고 다음 섹션은 56행에 시작하므로 전문 카드 본문을 건드리지 않고 독립 섹션을 삽입할 수 있다(`web/templates/landing.html:40-60`).

- 데스크톱 Top 3 가로 카드, 모바일 세로 목록.
- 각 항목은 순위, 기관, 짧은 카드 제목, `N명 반응 · 하트 H · 스크랩 S`, 상세 카드 딥링크만 표시한다.
- 원문·요약을 RPC 응답에서 복제하지 않는다. 링크는 `search-index.json`의 기존 `href`를 그대로 쓴다(`web/render.py:652-658`).
- 인기 랭킹은 랜딩 탐색 보조일 뿐이다. 브리프 안의 카드 순서, Evidence, Signal Tier, TL;DR, Findings 검색 결과를 재정렬하거나 수정하지 않는다.
- 네트워크·RPC 실패가 정적 사이트 열람을 막지 않는 progressive enhancement 원칙을 유지한다. 현행 반응층도 실패를 삼키고 정적 카드에 영향을 주지 않도록 설계돼 있다(`web/assets/reactions.js:1-5`, `web/templates/base.html:285-286`).

### 3.5 빈약 데이터와 장애 폴백

1. 알려진 카드가 2개 이상이면 실제 랭킹만 표시한다. N개를 채우려고 0점 카드를 인기 카드처럼 섞지 않는다.
2. 알려진 카드가 0~1개이거나 전체 `reactors`가 제품 임계값 미만이면 섹션 제목을 **`이번 주 새 카드`**로 바꾸고 최신 브리프의 결정론적 render order 상위 3개를 보여준다. 인기라는 표현과 순위 숫자를 제거한다.
3. RPC/네트워크/설정 실패도 같은 정적 최신 카드 폴백으로 수렴한다. 오류 문구나 빈 스켈레톤을 랜딩에 남기지 않는다.
4. 라이브 반응 분포가 `미확인`이므로 임계값은 구현 전에 anon read-only 계측으로 정한다. 기본 제안은 `알려진 카드 2개 이상 AND reactors 합계 3 이상`이며 확정값이 아니다.

## 4. 조사 B — 일일 수집 건수 소스 후보 전수 조사

## 5. 구르미 상태 위젯 서빙 경로 판정

### 5.1 안 A — 렌더타임 결정론 임베드

최신 커밋 브리프의 `coverage.intake_total`과 `coverage.rendered`를 `render.py`가 landing context로 넘기고 Jinja가 HTML에 직접 쓴다. 이미 동일 경로가 랜딩 이슈 카드와 하단 CTA에서 사용된다(`web/render.py:592-608`, `web/templates/landing.html:21-36`, `web/templates/landing.html:120-125`).

장점은 클라이언트 fetch 0, 새 RLS/RPC/secret 0, 정적 HTML과 골든 테스트로 재현 가능, Supabase 장애와 무관하다는 점이다. 한계는 최신 브리프가 바뀌는 주 1회만 값이 갱신된다는 것이다. intake는 매일 실행되지만(`.github/workflows/grm-intake.yml:1-14`) 웹 production 값은 브리프 JSON을 조립한 PR이 사람 머지된 뒤 배포된다(`.github/workflows/grm-web-publish.yml:5-15`, `.github/workflows/grm-web-deploy.yml:11-18`).

따라서 안 A에서는 `오늘`을 쓰지 않는다. **`이번 주 구르미가 규제 소식 N건을 먹고, 핵심 카드 M장으로 정리했어요 · YYYY-MM-DD 발행 기준`**처럼 기간과 기준일을 같이 보여준다.

### 5.2 안 B — 공개 카운트 RPC

`raw_signals`에 count-only `SECURITY DEFINER` RPC를 얹으면 클라이언트는 daily에 가까운 값을 받을 수 있다. 그러나 현재 행만으로 `scheduled intake`와 `historical backfill`을 구분할 수 없고, 백필이 과거 문서에도 현재 `collected_at`을 쓴다(§4). 따라서 단순 `오늘 ingested_at` RPC는 기술적으로 가능해도 제품 문구의 진실성을 충족하지 못한다.

정확한 `오늘 inserted`가 반드시 필요해지는 후속 단계에서는 다음처럼 **요약층을 먼저** 만들어야 한다.

1. 성공한 daily intake 끝에서 `grm-health.json.sources[].inserted` 합계를 계산한다.
2. service role만 쓸 수 있는 일일 요약 행에 `run_date_kst`, `inserted_total`, `generated_at_kst`, `health_status`를 멱등 upsert한다. historical backfill job은 이 표에 쓰지 않는다.
3. 공개 RPC는 최신 성공행의 `run_date_kst`, `inserted_total`, `generated_at_kst`만 반환한다. source별 수, 원문, 문서 ID, 오류 상세는 반환하지 않는다.
4. 오늘 실행 전·실패 시 0으로 가장하지 않고 마지막 성공 기준일을 함께 표시한다.

이는 `RPC 하나`가 아니라 새 요약 스키마와 daily workflow 쓰기까지 필요한 별도 기능이다. 현 단계에서는 표면과 운영부담이 v1 대중성 위젯의 가치보다 크다.

### 5.3 권장안

**v1은 안 A를 채택하고 신규 RPC를 만들지 않는다.** 구르미는 주간 발행 리듬을 친근하게 설명하는 장식·상태 요약으로만 쓴다. 제품이 정확한 당일 활성도를 검증한 뒤에만 안 B의 일일 요약층을 별도 설계·승인한다. `raw_signals` 직접 공개 또는 raw count를 `오늘 신규 규제`로 포장하는 안은 기각한다.

## 6. 구르미 문구·연출 3안

권장 위치는 히어로의 4개 통계 바 아래 또는 그와 한 덩어리인 작은 상태 칩이다(`web/templates/landing.html:40-53`). 인기 카드 랭킹은 그 아래 독립 섹션으로 두어 두 기능이 서로 경쟁하지 않게 한다. 브리프 상세, Findings, Evidence 배지, 시사점, 점검사항에는 구르미를 넣지 않는다.

| 안 | 문구 | 연출 | 평가 |
|---|---|---|---|
| A — 먹고 정리 | **`이번 주 구르미가 규제 소식 N건을 먹고, 핵심 카드 M장으로 정리했어요.`** `YYYY-MM-DD 발행 기준` | 작은 그릇의 종이 조각이 한 번 사라지고 카드 묶음이 나타나는 1회 모션 | **권장.** `intake_total`과 `rendered`의 실제 관계를 가장 정직하게 설명한다(`web/templates/landing.html:35`, `web/templates/brief.html:11`). |
| B — 이번 주 식단 | **`구르미의 이번 주 식단 · 규제 소식 N건 → 카드 M장`** | 그릇 옆 `N → M` 숫자 전환, 클릭 시 최신 브리프 | 짧고 모바일에 강하다. 다만 `식단`이 전문 사이트 톤보다 가벼울 수 있어 카피 검토 필요. |
| C — 모아 왔어요 | **`구르미가 이번 주 규제 소식 N건을 모아 왔어요. 핵심만 만나보세요.`** | 구름 주머니에서 문서 아이콘 2~3개가 나오는 정적 일러스트 + CTA | 가장 무난하다. `먹다` 표현을 부담스러워하는 사용자에게 적합하지만 `rendered` 수치 설명력은 낮다. |

공통 원칙은 다음과 같다.

- `확인 완료`, `검증 완료`, `안전`, `중요도`처럼 전문적 판단을 암시하는 말을 구르미에게 시키지 않는다. 위젯은 수집량과 이동 CTA만 담당한다.
- 수치는 텍스트로 제공하고 일러스트는 `aria-hidden` 처리한다. 자동 모션은 1회·짧게 하고 `prefers-reduced-motion`에서는 제거한다.
- 값과 함께 `발행 기준일`을 표시한다. 9일 이상 새 브리프가 없을 때는 `이번 주` 대신 `최근 발행 기준`으로 낮추는 폴백을 둔다.
- 구르미 상태가 실패해도 전문 콘텐츠, 카드 순서, 출처·원문 링크가 변하지 않는다. 정적 사이트의 progressive enhancement 원칙과 같다(`web/templates/base.html:285-286`).

## 7. 후속 구현 순서와 완료 게이트

이 문서는 구현 승인이 아니다. 승인 후에도 인기 카드와 구르미를 서로 다른 작은 변경으로 나누는 것이 안전하다.

### 7.1 인기 카드

1. 주간 의미·KST 경계·Top N·빈약 데이터 임계값을 제품 결정으로 확정한다.
2. count-only RPC migration과 SQL 계약 테스트를 만든다. user/raw/text/url 반환 부재를 자동 검사한다.
3. landing 정적 폴백 3개를 먼저 렌더하고, 반응 JS가 RPC 성공 시에만 인기 목록으로 교체하게 한다.
4. RPC 응답 ID를 `search-index.json`과 교차하고 알려지지 않은 ID를 버린다.
5. 익명 사용자·로그인 사용자·네트워크 실패·빈 데이터·동률·주 경계·toggle 반복을 검증한다.

완료 기준은 `가장 많이 본` 문구 0, 공개 응답의 개인·원문 필드 0, 한 계정의 카드당 인기점수 최대 1, 실패 시 정적 최신 카드 유지다.

### 7.2 구르미 위젯

1. 최신 brief coverage만 입력으로 쓰고 외부 fetch를 추가하지 않는다.
2. `이번 주`·발행 기준일·N/M을 함께 렌더한다.
3. 랜딩과 모바일 골든, reduced-motion, 스크린리더 텍스트를 검증한다.
4. 브리프·Findings·Evidence·Signal·전문 카피 byte를 변경하지 않았음을 회귀 검사한다.

완료 기준은 신규 RPC 0, 클라이언트 fetch 0, 주간 신선도 한계의 명시, 전문 콘텐츠 무개입이다.

## 8. 미확인·잔여 결정

- 라이브 Supabase에 `001_reaction.sql`/hardening이 어느 버전으로 적용됐는지, 현재 반응 행 수와 주간 분포는 확인하지 않았다. 이 세션은 라이브 Supabase를 조회하지 않았다.
- 라이브 Auth가 이메일 확인을 강제하는지와 다계정 생성 난이도는 저장소만으로 확인할 수 없다.
- 주간 RPC의 실제 latency·scan 행 수·필요 인덱스는 `EXPLAIN` 전까지 미확인이다. 선제 인덱스 추가 판정은 하지 않는다.
- 주간 경계를 `월 00:00 KST`로 할지 `최신 브리프 발행시각 이후`로 할지는 제품 결정이 필요하다. 본 문서는 사용자 문구와 주간 발행 리듬에 맞춰 월요일 KST를 권장한다.
- 인기 폴백 임계값(`2개 카드·3명`)은 라이브 분포 확인 전 제안값이다.
- 현재 커밋 브리프 129개에서 웹 ID 중복은 0이지만, `reaction.card_id`에 정적 카드 FK가 없고 웹 ID의 전역 유일성 DB 제약도 없다. 후속 구현에서 전체 `search-index.json` ID 중복 검사를 빌드 게이트로 둘지 결정해야 한다.
- 정확한 일일 위젯을 후속 추진할 경우 요약표 보존기간, 실패 run 처리, 재실행 upsert 우선순위, 공개 기준일 타임존을 별도 확정해야 한다.

## 9. 최종 RPC 필요 여부

- **인기 카드:** 권장한 엄밀한 주간 Top N에는 **신규 count-only RPC 필요**. 새 Edge Function·이벤트 표는 불필요하다. RPC 없이 가려면 의미와 문구를 누적 `이번 호 독자 반응`으로 낮춘다.
- **구르미 위젯:** 권장 v1에는 **신규 RPC 불필요**. 최신 브리프 coverage를 렌더타임에 임베드한다. 정확한 `오늘`을 요구하는 후속 버전은 raw count RPC만으로는 불충분하며 일일 health 요약층과 공개 count-only RPC가 함께 필요하다.
