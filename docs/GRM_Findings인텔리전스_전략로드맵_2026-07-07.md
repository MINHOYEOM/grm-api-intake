## 1. 전체 방향성

**한 줄 정의: "주간 규제 다이제스트" → "누적 규제 인텔리전스 자산 + AI 에이전트"**

정보 전달만으로는 지불 의사가 생기지 않는다(규제 정보 자체는 무료). 시장(Redica 200+ 고객, 톱20 제약사 중 19곳)이 증명한 공식은 **데이터 축적(해자) → 검색·분석 구독 → 업무 결합**이며, GRM은 이미 그 원자재(483 Observation·MFDS GMP 지적 표·WL 위반·WHOPIR excerpt의 결정론 추출)를 보유하고 있다.

### 3-덩어리 목표 아키텍처

```
① 수집·발행 파이프라인 (현행 유지·불변)
   GitHub Actions 수집기 → Notion Intake(주간 발행 큐) → Routine → 웹/뉴스레터
                    │
                    └─(신규: 이중 적재, additive)
                                ▼
② Findings 인텔리전스 자산 (신규 = 해자)
   raw_signals(전 수집물 무가공 보존) + findings(지적사항 정규화)
   파이프라인 내 SQLite → Supabase Postgres(서빙)
                                ▼ (PostgREST read-only API 공용)
③ 소비 인터페이스 3종
   웹 대시보드(검색·히트맵·업체이력·실사관 프로파일)
   뉴스레터(무료 유입 채널)
   Copilot Studio 에이전트(사내 Q&A·공모전 데모)
```

### 포지셔닝 (니치)

글로벌 플레이어(Redica·Cortellis·IQVIA)는 MFDS 커버리지가 약하고 전부 영어·연 수천만 원급. **"MFDS×글로벌 교차 + 한국어 + 저가"** 가 비어 있는 시장이며, 타깃은 실사 경험이 부족한 중소·중견 제약사의 생산·품질·RA.

## 2. 데이터 아키텍처

### 2.1 2층 스키마

- **`raw_signals`** — 수집된 모든 것(WL·회수·가이드라인·행정처분·483·GMP…)을 원본 JSON + 출처 URL + 수집일시 그대로 무가공 보존. 미래의 새 분석 기능은 재수집 없이 이 층에서 재추출.
- **`findings` (`grm-finding/v1`)** — 지적사항만 정규화한 분석층. 필드(안): 기관 · 문서유형 · 발행일 · 업체(정규화 entity) · 제조소 · 국가 · 제품군/모달리티 · 지적 카테고리 · 원문 인용 · 출처 URL · evidence · (483) 실사관.

### 2.2 정규화 3종 세트

1. **업체 별칭 사전** — "Hanmi Pharm Co., Ltd." = "한미약품(주)" entity 통합. 백필 과정에서 점진 구축.
2. **지적 카테고리 택소노미 15~20개** — 21 CFR 조항·MFDS 지적 문구를 공통 카테고리로 매핑(데이터완전성·무균보증·세척밸리데이션·일탈/CAPA·문서화·시설환경·원자재관리 등). 결정론 규칙+사전 1차, LLM 분류 보조(기존 verify 게이트 패턴), 커버리지 80% 목표.
3. **제품군** — 기존 모달리티 태그 재사용 + 제형 확장.

### 2.3 저장소 전략

| 층 | 기술 | 이유 |
|---|---|---|
| 파이프라인 내 | SQLite (+`findings.json` 산출) | GitHub Actions 정합·$0·결정론 빌드 유지 |
| 서빙 | Supabase Postgres (SQLite에서 동기화) | 테이블 생성 즉시 REST API(PostgREST) → 웹 대시보드와 Copilot Studio 커스텀 커넥터가 **같은 API** 사용 |
| Notion Intake | 현행 유지 — "주간 발행 큐"로 역할 한정 | 멱등성 기계장치 재작성 리스크 회피. 데이터 영구보존 책임에서 해방 |

**Notion→Postgres Intake 이전 트리거(공모전 이후 별도 트랙):** 다사용자 전환 / Notion API rate limit(3 req/s)이 수집 볼륨을 제약 / Routine 읽기 경로 재설계 시점. 그때는 Supabase가 findings 워크로드로 운영 검증된 상태.

### 2.4 기존 불변식과의 관계

- 콘텐츠 층(웹 카드: 정적·결정론·provenance·골든)은 **완전 불변**. Findings는 반응 층과 같은 "격리된 별도 층" 선례를 따름.
- 이중 적재는 additive — Notion 쓰기 로직·handoff·Status 흐름 무수정.

## 3. 5개월 로드맵 (M1~M5)

> **최상위 규칙: 스키마 동결 전에 백필을 시작하지 않는다.** (재작업 방지)

### M1 (7월) — 스키마 동결 + 내부 백필
- `grm-finding/v1` + `raw_signals` 스키마 확정·동결
- 지적 카테고리 택소노미 v1 (15~20개)
- **Notion 일회성 익스포트** → `raw_signals` (페이지 본문 원본 JSON 활용, 수천 row·API 한도 내 1일 작업) = 가장 싸고 확실한 "내부 백필"
- 수집기 **이중 적재 배선** (같은 실행에서 Supabase/SQLite append, additive)
- 기존 수집분 findings 변환 → 정적 검색 페이지 v0 (빌드타임 `findings.json`, `archive.js` 클라이언트 검색 선례 재사용)

### M2 (8월) — 외부 백필 (볼륨 확보)
- FDA 483 리딩룸 전수(~2,000건) · WL 아카이브 수년치 · MFDS GMP 과거 PDF
- 백필 파이프라인 재현 가능화: 실패 격리, OCR 불량 수동검토 큐, 원본 사이트 rate limit 준수
- 업체 별칭 사전 병행 구축 (백필 = 사전 만드는 과정)

### M3 (9월) — 서빙 층 + 분석 대시보드
- SQLite→Supabase 동기화 + read-only API (RLS·읽기전용 키)
- 웹 대시보드: 파셋 검색(기관×카테고리×제품군×연도) · 히트맵 · 업체 지적 이력 페이지 · **실사관 프로파일**(483 실사관별 지적 성향 — 같은 스키마에서 무료로 파생)

### M4 (10월) — Copilot 에이전트 + 사내 파일럿
- Copilot Studio 커스텀 커넥터 → findings API 연결
- Q&A 시나리오 5~10개 튜닝 (예: "작년 FDA가 무균주사제에서 가장 많이 지적한 항목은?" / "○○사 최근 3년 지적 이력")
- **생산·품질·RA 3부서 파일럿 2~4주** — 주간 브리프 회의 안건화 + 대시보드/에이전트 사용
- **효과 측정 병행** (파일럿 기간 = Before/After 데이터 수집 기간):
  - 수동 모니터링 대비 시간 절감 실측
  - 한 달치 재현율 테스트 (GRM이 잡은 신호 vs 놓친 신호)
  - EVAL 스코어카드 = 품질 지표
  - 파일럿 부서 실사용 코멘트

### M5 (11월~공모전) — 패키징
- 효과 정량화 정리 (Before/After 숫자·재현율·파일럿 증언)
- 데모 시나리오 리허설 (대시보드 + Copilot 에이전트)
- 리스크 서술 정리 (환각 0 결정론 층·provenance 하드 게이트·Evidence 등급·PII=SaaS 위임·저작권=원문 링크 방식)
- 발표 자료

- **F2d 정규화**: 업체 별칭 사전 v1(백필 과정에서 점진 구축, 커버리지 80% 목표)·OCR 불량 수동검토 큐.
- **F2e 공개·번역 정책 게이트(신규 설계 결정 필요)**: 백필분은 전량 영문 → 현행 공개 게이트(국문 필수)에 걸려 웹 비노출이며, 주간 번역(구독 한도)으로 수천 건 소화 불가. 채택안: **집계(트렌드·히트맵·카운트)는 번역 없이 전량 활용**(사전계산 stats 층은 원문 텍스트를 노출하지 않으므로 게이트와 무충돌), **row 노출은 최근 12개월 우선 번역**·과거분은 점진/미번역 유지. F3 설계의 전제.
- 완료 기준: findings ≥ 2,000건 · 연도 커버리지 ≥ 3년 · 백필 재현 가능(문서화된 재실행 경로) · 별칭 사전 커버리지 80%.

## F3. 인텔리전스 — 분석 서빙·대시보드 (8월 중순 ~ 9월 말) 〔원 M3〕

목표: "검색 DB" → "트렌드 인텔리전스" 전환(공모전 창의성 10점 + 확장성 일부).

- **F3c 업체 지적 이력 페이지**: entity 정규화(F2d) 기반.
- **F3d 실사관 프로파일**: 483 `inspector_names`(스키마 기보유) 기반 지적 성향.
